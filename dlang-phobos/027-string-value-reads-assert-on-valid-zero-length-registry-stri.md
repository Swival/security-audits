# Zero-length registry strings trigger assertion

## Classification
- Type: error-handling bug
- Severity: medium
- Confidence: certain

## Affected Locations
- `std/windows/registry.d:505`

## Summary
`regQueryValue(..., out string, ...)` treats `REG_SZ` and `REG_EXPAND_SZ` data as a UTF-16 buffer and asserts that the final code unit is a NUL terminator before checking whether the buffer is empty. When `cbData == 0`, which is valid for an empty registry string, the buffer slice is empty and the assertion fires. This causes reads of valid empty string registry values to abort under assertions instead of returning `""`.

## Provenance
- Verified from the provided reproducer and source analysis in `std/windows/registry.d`
- Scanner reference: https://swival.dev

## Preconditions
- Registry value type is `REG_SZ` or `REG_EXPAND_SZ`
- Stored value has zero-length data (`cbData == 0`)

## Proof
At `std/windows/registry.d:505`, string reads derive `wstr` from `cbData / wchar.sizeof` and immediately assert `wstr.length > 0 && wstr[$-1] == '\0'`. If `cbData` is `0`, `wstr.length` is `0`, so the assertion fails before the later guarded logic can return an empty string.

The reproduced path is practical and self-generated by the library:
- `Key.setValue(string name, string value, bool asEXPAND_SZ)` computes byte length with `lstrlenW(pszTmp) * wchar.sizeof` at `std/windows/registry.d:1077`
- For `""`, that length is `0`
- It passes `cbData == 0` to `RegSetValueExW` at `std/windows/registry.d:1079`
- A later read through `Value.value_SZ` reaches the failing assertion path

Windows permits `lpData == NULL` with `cbData == 0` for registry writes, so this is valid input, not registry corruption.

## Why This Is A Real Bug
The failure occurs on a valid Windows registry state and can be produced by this library's own write API. That makes the crash reachable in normal use, not only through malformed external data. The behavior is inconsistent across build modes: assertion-enabled builds abort, while assertion-disabled builds return `""`. This is a correctness and reliability bug in error handling around valid edge-case input.

## Fix Requirement
Handle `cbData == 0` for string registry values as an empty string before any terminator assertion, while preserving the existing terminator validation for non-empty buffers.

## Patch Rationale
The patch should short-circuit zero-length `REG_SZ` and `REG_EXPAND_SZ` reads to `""` before constructing an assertion that requires at least one UTF-16 code unit. This preserves the intended invariant for non-empty registry strings and aligns read behavior with valid Windows semantics and the library's own write behavior.

## Residual Risk
None

## Patch
- `027-string-value-reads-assert-on-valid-zero-length-registry-stri.patch` handles zero-length string data before the terminator assertion in `std/windows/registry.d`, so valid empty `REG_SZ` and `REG_EXPAND_SZ` values now return `""` instead of aborting under assertions.
