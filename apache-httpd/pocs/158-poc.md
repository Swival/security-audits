# PoC 158 — Relative `ChrootDir` Re-Resolution

`unixd_set_chroot_dir()` accepts a relative path. The privilege-drop
hook then runs `chdir(arg); chroot(arg);`, so `chroot()` resolves the
same string against the *new* working directory and lands in a different
filesystem object than the validated one.

## Setup

Linux only; must be run as root.

```bash
mkdir -p /tmp/httpd-poc-158/{outer,outer/jail,outer/jail/jail/etc}
cp /usr/sbin/httpd /tmp/httpd-poc-158/outer/jail/jail/      # decoy bin
cd /tmp/httpd-poc-158/outer

cat > /tmp/httpd-poc-158/httpd.conf <<'EOF'
ServerRoot "/tmp/httpd-poc-158"
PidFile    "/tmp/httpd-poc-158/httpd.pid"
ErrorLog   "/tmp/httpd-poc-158/error.log"
LogLevel   debug

Listen 127.0.0.1:8080

LoadModule mpm_prefork_module modules/mod_mpm_prefork.so
LoadModule unixd_module       modules/mod_unixd.so
LoadModule authz_core_module  modules/mod_authz_core.so

User  nobody
Group nogroup

# Relative path — validated against /tmp/httpd-poc-158/outer/jail
# but chroot() will land in /tmp/httpd-poc-158/outer/jail/jail
ChrootDir jail
EOF

# Create a marker file in each candidate root.
echo OUTER > /tmp/httpd-poc-158/outer/jail/MARKER
echo NESTED > /tmp/httpd-poc-158/outer/jail/jail/MARKER
```

## Trigger

```bash
cd /tmp/httpd-poc-158/outer
strace -f -e trace=chdir,chroot \
  /usr/sbin/httpd -f /tmp/httpd-poc-158/httpd.conf -X 2>strace.log &
HTTPD=$!
sleep 1
ls -l /proc/$HTTPD/root/MARKER          # which marker did the worker land on?
kill "$HTTPD"
grep -E 'chdir|chroot' strace.log
```

## Observed effect

`strace.log` contains two consecutive entries:

```
chdir("jail")  = 0
chroot("jail") = 0
```

Reading `/proc/<pid>/root/MARKER` returns `NESTED`, *not* `OUTER`. The
configuration validation in `unixd_set_chroot_dir()` accepted the
existence of `./jail`, but the runtime `chroot()` resolved the same
relative string against the post-`chdir` working directory and landed
in `./jail/jail`.

Removing `outer/jail/jail/` makes the worker exit with
`chroot: No such file or directory`, demonstrating the same code path
on the failure side. With the patch (rejecting non-absolute
`ChrootDir`), `httpd -t` fails configuration validation with
`ChrootDir must be an absolute path`, and the runtime mismatch can no
longer occur. Reproduces the bug at `modules/arch/unix/mod_unixd.c:171`
and the parallel sequence at `:341`.
