# PoC 057 — `mod_ext_filter` discards `ef_unified_filter()` failure

After logging an `ef_unified_filter()` error, `ef_output_filter()` falls
through to `ap_pass_brigade()`. When the downstream pass succeeds, the
filter returns `APR_SUCCESS`, hiding the upstream failure.

## Setup

Build with `mod_ext_filter`:

```
./configure --enable-ext-filter --with-mpm=event \
            --enable-mods-static=few
make -j4
```

External filter that closes stdin immediately, forcing
`pass_data_to_filter()` to fail when the next chunk is written:

```bash
cat > /tmp/badfilter.sh <<'SH'
#!/bin/sh
# Read just one byte then exit, leaving the pipe broken for further writes.
head -c 1 >/dev/null
exit 0
SH
chmod +x /tmp/badfilter.sh
```

`httpd.conf`:

```
Listen 127.0.0.1:8080
LoadModule ext_filter_module modules/mod_ext_filter.so
LoadModule mime_module       modules/mod_mime.so
ServerName localhost

ExtFilterDefine bad mode=output cmd="/tmp/badfilter.sh"

DocumentRoot /var/www
<Directory /var/www>
    SetOutputFilter bad
    Require all granted
</Directory>
```

Create a payload large enough that the filter receives multiple writes:

```
yes ABCDEFGH | head -c 65536 > /var/www/big.txt
```

## Trigger

```
curl -i http://127.0.0.1:8080/big.txt -o /tmp/out.body
```

Race driver to amplify the timing where `head -c 1` exits before all
brigade chunks are written:

```
for i in $(seq 1 100); do
    curl -s http://127.0.0.1:8080/big.txt -o /dev/null &
done
wait
```

## Observed effect

`error_log` records the filtering failure:

```
[ext_filter:error] AH01468 ef_unified_filter() failed
[ext_filter:debug] AH01462 writing data to filter
[ext_filter:debug] write to filter: Broken pipe
```

But the access log still reports `200` and the response body is whatever
partial data made it through before the broken pipe. Pre-patch
`ef_output_filter()` returns `APR_SUCCESS`; the request handler observes
no failure, the response status is 200, and the client receives a
truncated body.

Post-patch the same trigger produces a 500 response (or the connection is
aborted) because the original filter error is propagated to the caller.

## Log marker

Pre-patch sequence (per request):

```
AH01468 ef_unified_filter() failed
... access_log: "GET /big.txt HTTP/1.1" 200 <bytes-written>
```

The `200` paired with the `AH01468` error is the observable inconsistency
that demonstrates the swallowed failure.
