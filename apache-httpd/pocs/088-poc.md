# PoC 088 - h2_push `ceil_power_of_2()` Signed Overflow

For `n > 1073741824`, `ceil_power_of_2()` performs `++n` on `INT_MAX`, which
is signed integer overflow (UB). Downstream, the negative size corrupts diary
sizing logic.

## Setup

This is a numeric/UB bug in `modules/http2/h2_push.c`. Reaching it from network
traffic requires `H2PushDiarySize` past the configuration cap (the in-tree
clamp keeps normal `httpd.conf` use safe), so the cleanest reproduction is a
direct unit harness against the affected helpers.

`tmp/ceil_pow2.c`:

```c
#include <stdio.h>
#include <apr.h>

static apr_int32_t ceil_power_of_2(apr_int32_t n) {
    --n;
    n |= n >> 1;
    n |= n >> 2;
    n |= n >> 4;
    n |= n >> 8;
    n |= n >> 16;
    return ++n;
}

int main(void) {
    apr_int32_t inputs[] = { 1, 1024, 1073741824, 1073741825, 2000000000 };
    for (size_t i = 0; i < sizeof(inputs)/sizeof(inputs[0]); i++) {
        printf("%d -> %d\n", inputs[i], ceil_power_of_2(inputs[i]));
    }
    return 0;
}
```

Build with UBSan to make the overflow explicit:

```bash
cc -O1 -g -fsanitize=undefined -I srclib/apr/include \
   tmp/ceil_pow2.c -o tmp/ceil_pow2
```

## Trigger

```bash
./tmp/ceil_pow2
```

For the in-tree path, build httpd with the sanitizer and call
`h2_push_diary_create` directly from a small driver, or set:

```apache
LoadModule http2_module modules/mod_http2.so
H2PushDiarySize 2000000000        # only effective if the clamp is bypassed
```

The configuration parser caps this in `h2_config.c:873`, so the public-facing
trigger requires reaching `h2_push_diary_create()` or
`h2_push_diary_digest_get()` with an unclamped value (other in-process module,
test code, or future caller).

## Observed effect

UBSan output:

```text
1 -> 1
1024 -> 1024
1073741824 -> 1073741824
modules/http2/h2_push.c:563:13: runtime error: signed integer overflow:
  2147483647 + 1 cannot be represented in type 'int'
1073741825 -> -2147483648
2000000000 -> -2147483648
```

`diary_create()` then computes:

```text
diary->NMax = -2147483648
diary->N    = -2147483648
```

`h2_push_diary_append()` evaluates
`diary->entries->nelts >= diary->N` against a negative limit; the condition is
permanently true, evicting on every append, or driving an infinite loop in
neighbouring eviction code paths depending on how the negative value is used
in arithmetic.

## Log marker

UBSan diagnostic line cited above. No httpd `error_log` marker on unsanitized
builds; the corruption is silent.

## Notes

- The patch adds `N <= (1 << 30)` to the `diary_create()` guard and
  `maxP > (1 << 30)` rejection to `h2_push_diary_digest_get()`. Both bound
  inputs to the largest value whose next power-of-two fits in signed 32-bit.
- A purely network-driven PoC would require an `H2PushDiarySize` value that
  bypasses the configuration cap; this is unreachable in stock 2.4.66 and is
  why the finding emphasises the helpers' callers rather than the SETTINGS
  frame path.
