# 034 - mod_cgid sock_writev reports short vectored writes as success

`modules/generators/mod_cgid.c:479` calls `writev()` (or `sendmsg()` on
fd-passing builds) once and returns `APR_SUCCESS` for any non-negative
return value. A positive partial write leaves the daemon reading a
truncated request stream.

## Setup

Same general environment as PoC 033. The cgid socket protocol uses a
fixed `cgid_req_t` followed by length-prefixed strings, so a short
write that lands inside the variable-length region misframes the next
read. We reproduce in two ways:

1. Standalone harness for `sock_writev()` driving a `socketpair`.
2. End-to-end via httpd with an `LD_PRELOAD` that returns a short
   `writev()` count for the first call only.

```bash
mkdir -p tmp/cgi-bin
printf '#!/bin/sh\nA=$(env | wc -c)\necho Content-Type: text/plain\n' \
  > tmp/cgi-bin/many-env.cgi
printf 'echo\necho env-bytes=$A\n' >> tmp/cgi-bin/many-env.cgi
chmod +x tmp/cgi-bin/many-env.cgi
```

The CGI script is intentionally trivial; the goal is to grow the
environment block enough that `send_req()` builds a multi-iovec call.
Inflate it from the client side:

```bash
HDRS=()
for i in $(seq 1 200); do HDRS+=( -H "X-Pad-$i: $(printf 'A%.0s' {1..200})" ); done
```

## Trigger

Harness:

```c
/* tmp/sock_writev_harness.c */
#include <sys/socket.h>
#include <sys/uio.h>
#include <unistd.h>
#include <string.h>
#include <stdio.h>

int main(void) {
    int sv[2]; socketpair(AF_UNIX, SOCK_STREAM, 0, sv);
    int sndbuf = 8192;
    setsockopt(sv[0], SOL_SOCKET, SO_SNDBUF, &sndbuf, sizeof(sndbuf));
    char a[1<<20], b[1<<20]; memset(a,'A',sizeof a); memset(b,'B',sizeof b);
    struct iovec v[2] = { {a,sizeof a}, {b,sizeof b} };
    ssize_t rc;
    do { rc = writev(sv[0], v, 2); } while (rc < 0 && errno == EINTR);
    /* Buggy helper would now return APR_SUCCESS unconditionally. */
    char rb[1<<16]; ssize_t got = recv(sv[1], rb, sizeof rb, MSG_DONTWAIT);
    printf("writev rc=%zd delivered=%zd of %zu\n", rc, got,
           sizeof a + sizeof b);
}
```

```bash
cc -O0 -g tmp/sock_writev_harness.c -o tmp/sock_writev_harness
./tmp/sock_writev_harness
```

Preload shim that forces the short partial:

```c
/* tmp/short_writev_preload.c */
#define _GNU_SOURCE
#include <dlfcn.h>
#include <sys/uio.h>
static ssize_t (*real_writev)(int, const struct iovec *, int);
ssize_t writev(int fd, const struct iovec *v, int n) {
    if (!real_writev) real_writev = dlsym(RTLD_NEXT, "writev");
    /* Force a short write inside the first iov on the first call. */
    static int once = 0;
    if (!once && n > 0 && v[0].iov_len > 8) {
        once = 1;
        return real_writev(fd, &(struct iovec){v[0].iov_base, 8}, 1);
    }
    return real_writev(fd, v, n);
}
```

```bash
cc -shared -fPIC tmp/short_writev_preload.c -ldl -o tmp/shortwv.so
LD_PRELOAD=$PWD/tmp/shortwv.so ./httpd -X -f $PWD/tmp/httpd.conf &
curl -sS "${HDRS[@]}" http://127.0.0.1:8080/cgi-bin/many-env.cgi
```

## Observed effect

Harness output:

```
writev rc=8192 delivered=8192 of 2097152
```

The buggy `sock_writev()` would treat that as `APR_SUCCESS`. End to
end, the cgid daemon's `get_req()` then reads truncated request bytes,
which manifests as one of:

- HTTP `500 Internal Server Error` and an
  `AH01258: cgid daemon failed to read request` entry in the error
  log.
- The connection hangs until `Timeout` while the daemon waits for
  trailing environment bytes that never arrive.
- Garbled CGI environment when the truncation lands at a string
  boundary, observable in the response body of `many-env.cgi`.
