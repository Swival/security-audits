# 015 — Worker MPM transaction-pool leak on queue failure

When `ap_queue_push_socket()` fails after `accept()`, the worker MPM listener
closes the socket and logs the error but never destroys or recycles
`ptrans`. The pool is overwritten on the next loop iteration and leaks for
the lifetime of the child process.

## Setup

```apache
LoadModule mpm_worker_module modules/mod_mpm_worker.so

ServerLimit         1
StartServers        1
ThreadsPerChild     2
MaxRequestWorkers   2

# Keep the queue tiny so it overflows quickly under load.
ListenBacklog 8
```

Force `ap_queue_push_socket()` to fail. The simplest deterministic way is
to patch the function to return `APR_EGENERAL` on every Nth call so the
leak is observable without a complex race; alternatively run with a debug
hook that interposes on the symbol via `LD_PRELOAD`:

```c
/* preload_fail_push.c -- gcc -shared -fPIC -o fail_push.so preload_fail_push.c -ldl */
#include <stdint.h>
#include <dlfcn.h>
typedef int (*fn_t)(void *q, void *csd, void *baton, void *ptrans);
static fn_t real;
int ap_queue_push_socket(void *q, void *csd, void *baton, void *ptrans) {
    static int n;
    if (!real) real = dlsym(RTLD_NEXT, "ap_queue_push_socket");
    if ((n++ & 0x1f) == 0) return 70008; /* APR_EGENERAL */
    return real(q, csd, baton, ptrans);
}
```

```bash
gcc -shared -fPIC -o fail_push.so preload_fail_push.c -ldl
LD_PRELOAD=$PWD/fail_push.so httpd -X &
HTTPD_PID=$!
```

## Trigger

Hammer the listener with short-lived connections so the failure injection
runs frequently:

```bash
for i in $(seq 1 20000); do
    curl -s -o /dev/null http://127.0.0.1/ &
done
wait
```

## Observed effect

Each injected failure logs:

```text
[mpm_worker:crit] AH03138: ap_queue_push_socket failed
```

RSS of the child process climbs without bound. Compare:

```bash
ps -o rss= -p $HTTPD_PID
```

Before run: ~5 MB. After 20 000 requests with 1/32 injected failures:
~120 MB and rising.

`pmap -X $HTTPD_PID` shows a growing number of anonymous APR allocator
mappings. Inspecting `ptrans` lifetime with gdb:

```text
(gdb) break worker.c:691
(gdb) commands
> printf "leaked ptrans=%p\n", ptrans
> continue
> end
(gdb) continue
leaked ptrans=0x55b3c1a0a040
leaked ptrans=0x55b3c1a14080
leaked ptrans=0x55b3c1a1e0c0
...
```

None of these pointers reappear in `ap_queue_info_pop_pool()` because they
were never returned to the idle list — confirming the leak.
