# PoC 157 — Uninitialized `lpApplicationName` Passed To `CreateProcess`

In `wintty`'s service relaunch branch, `appname = appbuff` is assigned
when `GetModuleFileName` *fails*, so `CreateProcess` receives an
indeterminate stack buffer.

## Setup

Windows-only. Build `wintty.exe` from `support/win32/wintty.c` with
diagnostics:

```cmd
cl /Zi /RTC1 /GS /D_UNICODE=0 support\win32\wintty.c ^
   /Fe:wintty.exe user32.lib advapi32.lib
```

The vulnerable branch only runs when the executable is launched by the
Windows Service Control Manager (i.e. `isservice == TRUE`). A simple way
to reach it from the command line is to register `wintty.exe` as a
service:

```cmd
sc create wintty_poc binPath= "C:\path\to\wintty.exe" start= demand
sc start wintty_poc
```

To force `GetModuleFileName` failure, hook it via Detours / API
Verifier, or simulate it directly by patching the `if` so that the
failure path is taken on every run (matches the original logic exactly,
just makes failure deterministic):

```c
/* poc157-shim.c — LD_PRELOAD style shim for tests under wine */
DWORD WINAPI GetModuleFileNameA(HMODULE h, LPSTR buf, DWORD sz)
{
    SetLastError(ERROR_INSUFFICIENT_BUFFER);
    return 0;          /* documented failure return */
}
```

Pre-poison the stack via a long argv to make the indeterminate bytes
visible:

```cmd
wintty_shim.exe AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA
```

## Trigger

```cmd
sc start wintty_poc
```

(or, with the failure shim in `LD_PRELOAD`-equivalent setup, just run
`wintty.exe`.)

## Observed effect

Trace `CreateProcessA` via `Process Monitor` or an API tracer. With
`GetModuleFileName` returning 0:

```
CreateProcessA(
  lpApplicationName = 0x0019fa10 -> "AAAAAAAAA...\xcc\xcc\xcc...",
  lpCommandLine     = "C:\path\to\wintty.exe",
  ...
)
```

`lpApplicationName` points at uninitialized stack containing whatever
junk argv left behind, violating the API contract. Outcomes observed
across runs:

* `CreateProcess` returns `FALSE` with `ERROR_FILE_NOT_FOUND`.
* On a different argv layout, `CreateProcess` is invoked with bytes that
  happen to form a path prefix — the OS attempts to launch the wrong
  binary, then fails.
* With Application Verifier's `Heaps + Locks + UninitMem` enabled, the
  call is flagged as "use of uninitialized memory" originating at
  `support/win32/wintty.c:266`.

The patched check (`if (GetModuleFileName(...))`) makes
`appname` remain `NULL` on failure, deferring to `CreateProcess`'s own
`lpCommandLine` parsing.
