# PoC 171 — Unfinished Multi Requests Report Success (mod_md curl)

`md_curl_multi_perform()`'s cleanup loop calls `fire_status(req, APR_SUCCESS)`
for every request still queued at the `leave:` label, even when the
dispatcher itself exits with an error. Status-only callbacks therefore
observe success for work that did not complete.

## Setup

Library: `mod_md` curl multi backend. The bug is exercised most cleanly in
a unit driver linked against `md_curl.o` and friends; an end-to-end test
needs an ACME flow that pushes multiple requests concurrently.

### Driver harness

`/tmp/171/poc171.c`:

```c
#include "md_http.h"
#include "md_curl.h"

static apr_status_t reqs_status[8];
static int reqs_n;

static apr_status_t on_status(const md_http_request_t *r,
                              apr_status_t s, void *ud) {
    reqs_status[reqs_n++] = s;
    return APR_SUCCESS;
}

int main(void) {
    apr_initialize();
    apr_pool_t *p; apr_pool_create(&p, NULL);
    md_http_t *http; md_http_create(&http, p, "poc/0.1", NULL);
    md_http_use_implementation(md_curl_get_impl(p));

    /* Queue several GETs. Use a TCP host that always RSTs so curl returns
       CURLE_COULDNT_CONNECT for every easy handle. 127.0.0.1:1 is fine. */
    apr_array_header_t *reqs =
        apr_array_make(p, 4, sizeof(md_http_request_t*));
    for (int i = 0; i < 4; i++) {
        md_http_request_t *req;
        md_http_GET_create(&req, http, "http://127.0.0.1:1/");
        md_http_set_on_status_cb(req, on_status, NULL);
        APR_ARRAY_PUSH(reqs, md_http_request_t*) = req;
    }

    apr_status_t rv = md_curl_multi_perform(http, p, reqs);
    fprintf(stderr, "rv=%d\n", rv);
    for (int i = 0; i < reqs_n; i++)
        fprintf(stderr, "callback[%d] status=%d\n", i, reqs_status[i]);
    return 0;
}
```

To force the multi dispatcher to exit through `leave` with `rv != 0`,
either:

- inject a `curl_multi_*` failure (e.g. `LD_PRELOAD` shim returning
  `CURLM_OUT_OF_MEMORY` on the second `curl_multi_add_handle`), or
- set `MDHttpProxy http://127.0.0.1:1` so `internals_setup` succeeds but
  `nextreq` later fails midway.

## Trigger

```sh
cc -I include -I srclib/apr/include -I modules/md \
   /tmp/171/poc171.c modules/md/md_curl.o modules/md/md_http.o ... \
   -o /tmp/171/poc171
LD_PRELOAD=/tmp/171/curlm_fail.so /tmp/171/poc171
```

## Observed effect

`stderr` shows the dispatcher returning a non-zero error while the
status callbacks all received `APR_SUCCESS`:

```text
rv=70008                       # CURLM_OUT_OF_MEMORY mapped to APR error
callback[0] status=0           # APR_SUCCESS
callback[1] status=0
callback[2] status=0
callback[3] status=0
```

This contradicts the documented contract for `md_http_request_t` status
callbacks: "called with `APR_SUCCESS` only when the operation completed".

Real-world impact in mod_md: the ACME renewal loop reads only
`fire_status` results for status-only requests, marks the order as
processed, and silently moves on without refreshing the certificate.
Subsequent `MDCheckInterval` cycles repeat the same broken renewal,
hiding the underlying error from operators.

After the patch (`fire_status(req, rv);`) the callbacks all receive
the dispatcher's actual error status:

```text
rv=70008
callback[0] status=70008
callback[1] status=70008
...
```

and the renewal loop correctly logs and retries.
