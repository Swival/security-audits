# PoC 126 - Wrong filter handle checked before KEEP_BODY insertion

`ap_request_insert_filter()` checks for `KEPT_BODY` but inserts `KEEP_BODY`,
so a duplicate `KEEP_BODY` input filter is added when `KeptBodySize` is on
and the request already carries a `KEEP_BODY` filter.

## Setup

Lab httpd 2.4.66 with `mod_request` and any module that consumes the
request body (we use a CGI sink). The configuration both forces a
`KEEP_BODY` filter via `SetInputFilter` and enables `KeptBodySize`.

`conf/httpd.conf`:

```
ServerName localhost
Listen 127.0.0.1:8080
LoadModule mpm_event_module    modules/mod_mpm_event.so
LoadModule authz_core_module   modules/mod_authz_core.so
LoadModule request_module      modules/mod_request.so
LoadModule cgid_module         modules/mod_cgid.so
LoadModule mime_module         modules/mod_mime.so
LoadModule unixd_module        modules/mod_unixd.so
LoadModule log_config_module   modules/mod_log_config.so

ErrorLog logs/error_log
LogLevel request:trace8 core:trace6
ScriptSock cgisock
AddHandler cgi-script .cgi
DocumentRoot "/tmp/htdocs"
<Directory "/tmp/htdocs">
    Require all granted
    Options +ExecCGI
    SetInputFilter KEEP_BODY
    KeptBodySize 65536
</Directory>
```

`/tmp/htdocs/echo.cgi`:

```sh
#!/bin/sh
echo "Content-Type: text/plain"
echo
cat
```

To make the duplicate filter visible, add a one-line debug to
`mod_request.c::ap_request_insert_filter()` for the lab build:

```c
ap_log_rerror(APLOG_MARK, APLOG_INFO, 0, r,
              "lab: inserting KEEP_BODY (existing filters: %s)",
              ap_get_input_filter_handle("KEEP_BODY") ? "yes" : "no");
```

## Trigger

```
curl -v --data 'hello=world' http://127.0.0.1:8080/echo.cgi
```

`mod_log_forensic` or the trace log will show two `KEEP_BODY` insertions.

A simpler proof without code change is to dump the input-filter chain with
`mod_dumpio`:

```
LoadModule dumpio_module modules/mod_dumpio.so
DumpIOInput On
LogLevel dumpio:trace7
```

The trace log shows `mod_request.c(...)` running `KEEP_BODY` twice for the
same request before the body is consumed.

## Observed effect

- Two `KEEP_BODY` filter records are linked into the request input chain.
- The first invocation populates `r->kept_body`; the second invocation
  observes that `r->kept_body` is already set and self-removes.
- Until the body is read, both filters remain on the chain. For requests
  that never read the body (auth-only fixups, redirects), the duplicate
  filter persists and consumes a per-request allocation.
- `LogLevel ... trace8` shows successive `ap_filter_input_filter_register`
  for the same handle.

With the patch (`request_is_filter_present(r, keep_body_input_filter_handle)`),
the second insertion is suppressed and only one `KEEP_BODY` record appears.
