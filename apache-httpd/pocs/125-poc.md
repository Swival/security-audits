# PoC 125 - Unchecked apr_bucket_read in inflate_out_filter

`mod_deflate`'s `INFLATE` output filter consumes `data`/`len` from
`apr_bucket_read()` without checking the status, so a failing CGI/proxy
response bucket can feed garbage into zlib or `memcpy()`.

## Setup

Lab httpd 2.4.66 with `mod_deflate` and `mod_cgid`, with the `INFLATE` output
filter explicitly inserted. The CGI script declares `Content-Encoding: gzip`,
emits a small valid prefix, then exits abnormally so the upstream CGI bucket
read fails after the gzip header has already been parsed.

`conf/httpd.conf`:

```
ServerName localhost
Listen 127.0.0.1:8080
LoadModule mpm_event_module    modules/mod_mpm_event.so
LoadModule authz_core_module   modules/mod_authz_core.so
LoadModule deflate_module      modules/mod_deflate.so
LoadModule cgid_module         modules/mod_cgid.so
LoadModule mime_module         modules/mod_mime.so
LoadModule unixd_module        modules/mod_unixd.so
LoadModule log_config_module   modules/mod_log_config.so

ErrorLog logs/error_log
LogLevel deflate:trace4
ScriptSock cgisock
TypesConfig conf/mime.types
AddHandler cgi-script .cgi

DocumentRoot "/tmp/htdocs"
<Directory "/tmp/htdocs">
    Require all granted
    Options +ExecCGI
    SetOutputFilter INFLATE
</Directory>
```

`/tmp/htdocs/bad.cgi` (valid gzip header, then dies mid-stream):

```sh
#!/bin/sh
printf 'Content-Type: text/plain\r\n'
printf 'Content-Encoding: gzip\r\n\r\n'
# 10-byte gzip header + 1 truncated deflate byte, then SIGKILL self
python3 -c 'import sys; sys.stdout.buffer.write(bytes.fromhex("1f8b0800000000000003ab"))'
kill -9 $$
```

`chmod +x /tmp/htdocs/bad.cgi`. Run httpd under valgrind:

```
valgrind --track-origins=yes ./httpd -X -d $PWD
```

## Trigger

```
curl -v --raw http://127.0.0.1:8080/bad.cgi >/dev/null
```

## Observed effect

After the gzip header bucket has been consumed, `inflate_out_filter()` issues
`apr_bucket_read()` against the next CGI bucket. The script's abrupt
termination causes the CGI bucket read to return a non-success status, but
`data`/`len` are used anyway:

- `error_log` shows `AH01387: inflate error -3` or `AH01386` with a length
  that exceeds the bytes actually transferred from the CGI helper.
- `valgrind --track-origins=yes` reports
  `Conditional jump or move depends on uninitialised value(s)` originating in
  `inflate_out_filter` near the `len > APR_INT32_MAX` test, then a tainted
  `memcpy` length when the stale `len` is used to copy into the output
  brigade.
- The client receives a partial response followed by a connection drop. The
  worker keeps running but the gzip stream is left in `Z_DATA_ERROR`.

With the patch applied, `inflate_out_filter()` propagates the read status
immediately and the same trigger ends with a clean `502` (or premature EOF)
without uninitialised reads.
