# 035 - mod_cgid SSI cleanup with uninitialized PID

`modules/generators/mod_cgid.c:1707` registered `cleanup_script` with an
uninitialized `info->pid` when `get_cgi_pid()` failed during SSI
`exec cmd` handling. Pool cleanup later passed that indeterminate value
to `kill()`.

## Setup

Minimum config to drive the SSI cmd path through `mod_cgid`:

```apache
# tmp/httpd.conf
ServerRoot   /tmp/apache-httpd-2.4.66
Listen       127.0.0.1:8080
LoadModule   mpm_event_module modules/mod_mpm_event.so
LoadModule   authz_core_module modules/mod_authz_core.so
LoadModule   mime_module modules/mod_mime.so
LoadModule   include_module modules/mod_include.so
LoadModule   cgid_module modules/mod_cgid.so
ErrorLog     /tmp/poc-err.log
LogLevel     debug
ScriptSock   /tmp/cgisock
DocumentRoot /tmp/poc-www
<Directory /tmp/poc-www>
  Options +Includes +ExecCGI
  AddType text/html .shtml
  AddOutputFilter INCLUDES .shtml
  Require all granted
</Directory>
```

```bash
mkdir -p /tmp/poc-www
printf '<!--#exec cmd="echo hi"-->\n' > /tmp/poc-www/x.shtml
```

## Trigger

`get_cgi_pid()` fails when the cgid daemon does not respond with a
valid PID. The simplest way to provoke that is to wedge the daemon
mid-exchange with a preload that drops the PID-reply read on the
handler side:

```c
/* tmp/getpid_fail_preload.c */
#define _GNU_SOURCE
#include <dlfcn.h>
#include <unistd.h>
#include <errno.h>
static ssize_t (*real_read)(int, void *, size_t);
ssize_t read(int fd, void *b, size_t n) {
    if (!real_read) real_read = dlsym(RTLD_NEXT, "read");
    /* Cause sock_read to fail inside get_cgi_pid for SSI requests
     * by returning EPIPE for the first read on any AF_UNIX stream. */
    static int once = 0;
    if (!once) { once = 1; errno = EPIPE; return -1; }
    return real_read(fd, b, n);
}
```

```bash
cc -shared -fPIC tmp/getpid_fail_preload.c -ldl -o tmp/getpidfail.so
LD_PRELOAD=$PWD/tmp/getpidfail.so ./httpd -X -f $PWD/tmp/httpd.conf &
curl -sS http://127.0.0.1:8080/x.shtml
```

For deterministic observation without the preload, add a one-line
`fprintf` near `cleanup_nonchild_process()` to print the PID it is
about to signal, rebuild, and rerun:

```diff
@@ static apr_status_t cleanup_nonchild_process(...)
+    fprintf(stderr, "POC cleanup_nonchild kill pid=%d\n", (int)pid);
     kill(pid, SIGTERM);
```

## Observed effect

With the failure path triggered, `info->pid` is whatever bytes
`apr_palloc()` happened to return. On glibc malloc that is typically
the previous allocation's contents; in pool memory the field commonly
ends up `0` or a small leftover value.

Expected log markers:

```
[debug] error determining cgi PID (for SSI)
POC cleanup_nonchild kill pid=0          # or another stale value
```

`kill(0, SIGTERM)` signals the entire process group of the worker,
which in `httpd -X` foreground mode terminates the server process
itself. With a non-zero stale PID, the worker issues `SIGTERM` and
later `SIGKILL` against an unrelated process owned by the same uid.
The exact target depends on heap contents; the bug class is the
indeterminate signal target on the failure path.
