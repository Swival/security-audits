# 022 — mod_isapi leaks write lock on DLL load failure

`isapi_lookup()` at `modules/arch/win32/mod_isapi.c:447` only releases
`(*isa)->in_progress` on success. A failed DLL load leaves the rwlock
permanently write-held, deadlocking every later request for the same DLL.

## Setup

Platform constraint: Windows (mod_isapi is Win32-only). Build httpd with
`mod_isapi` enabled, e.g. via the Apache Win32 NMake build:

```cmd
nmake /f Makefile.win _apacher INSTDIR=C:\Apache24
```

`httpd.conf` snippet:

```apache
LoadModule isapi_module modules/mod_isapi.so

<Directory "C:/Apache24/htdocs/isapi">
    SetHandler isapi-handler
    Options +ExecCGI
    Require all granted
</Directory>
```

Place a deliberately broken ISAPI DLL so `LoadLibraryEx` succeeds but
`GetProcAddress("HttpExtensionProc")` fails:

```c
/* broken_isapi.c - build with: cl /LD broken_isapi.c */
#include <windows.h>
BOOL WINAPI DllMain(HINSTANCE h, DWORD r, LPVOID v) { return TRUE; }
/* deliberately no GetExtensionVersion / HttpExtensionProc exports */
```

```cmd
cl /LD broken_isapi.c
copy broken_isapi.dll C:\Apache24\htdocs\isapi\broken.dll
```

`isapi_load()` will fail with `APR_EMISMATCH` / no entry points.

## Trigger

Drive the same DLL from two clients. The first request enters
`isapi_lookup()` with `r != NULL`, takes the write lock, then `isapi_load()`
fails. The unlock branch is never taken. The second request blocks forever
waiting for the read or write lock.

```cmd
:: Window 1
curl -v http://127.0.0.1/isapi/broken.dll

:: Window 2 (after Window 1 returns 500)
curl -v --max-time 30 http://127.0.0.1/isapi/broken.dll
```

A concurrent driver to make the deadlock obvious:

```bash
for i in $(seq 1 8); do
    curl -s -o /dev/null --max-time 60 \
        "http://127.0.0.1/isapi/broken.dll" &
done
wait
```

## Observed effect

- Window 1 returns 500 with `error.log` line `mod_isapi: ... failed to load`.
- Window 2 (and every subsequent request to `/isapi/broken.dll`) hangs until
  `--max-time` fires. `error.log` shows no further activity for that DLL.
- `procdump -ma httpd.exe` of the worker shows a thread parked in
  `apr_thread_rwlock_rdlock` / `wrlock` referencing the cached
  `isapi_loaded` entry; the rwlock state shows a writer count of 1 with no
  owner thread still inside `isapi_lookup`.
- After `ISAPI_RETRY` (default 30s) elapses, the retry path tries to take
  the write lock and also blocks indefinitely.

After the patch is applied, Window 2 returns 500 promptly and the rwlock is
not held between requests.
