# PoC 071 - Zero-Capacity h2_iqueue Modulo Crash

`h2_iq_create(pool, 0)` leaves `q->nalloc == 0`. The first
`h2_iq_add()`/`h2_iq_append()` reaches `i = (q->head + q->nelts) %
q->nalloc` and divides by zero - on x86 a `SIGFPE`, on aarch64 undefined
behavior with similar abort.

## Setup

Vulnerable httpd 2.4.66 source tree. Because no in-tree caller currently
passes `0` (`h2_conf_set_max_streams()` rejects values `< 1`), this PoC
exercises the public queue API directly through a tiny test driver linked
against the httpd build tree, demonstrating that the API admits the
invalid state.

```
cd /tmp/apache-httpd-2.4.66
./configure --with-included-apr --enable-http2 --prefix=/tmp/httpd-iq
make -j
```

Driver `iq_zero.c`:

```c
#include <apr_pools.h>
#include <stdio.h>
#include "modules/http2/h2_util.h"

int main(void) {
    apr_initialize();
    apr_pool_t *p; apr_pool_create(&p, NULL);

    h2_iqueue *q = h2_iq_create(p, 0);
    fprintf(stderr, "created with capacity 0, about to append...\n");
    h2_iq_append(q, 1);                /* boom */
    fprintf(stderr, "should not reach here\n");
    return 0;
}
```

Build and link against the in-tree object:

```
gcc -I /tmp/apache-httpd-2.4.66/include \
    -I /tmp/apache-httpd-2.4.66/srclib/apr/include \
    -I /tmp/apache-httpd-2.4.66/modules/http2 \
    -o /tmp/iq_zero /tmp/iq_zero.c \
    /tmp/apache-httpd-2.4.66/modules/http2/.libs/h2_util.o \
    /tmp/apache-httpd-2.4.66/srclib/apr/.libs/libapr-1.a \
    -lpthread -ldl
```

Platform note: on macOS use `-Wl,-no_pie` and pull APR via `pkg-config
apr-1`; on Linux statically linked APR works as shown.

## Trigger

```
/tmp/iq_zero
```

## Observed effect

```
created with capacity 0, about to append...
Floating point exception (core dumped)
```

`gdb` on the core file places the fault inside `h2_iq_add`:

```
Program terminated with signal SIGFPE, Arithmetic exception.
#0  h2_iq_add (q=..., sid=1, ...)
    at modules/http2/h2_util.c:366
        i = (q->head + q->nelts) % q->nalloc;
(gdb) p q->nalloc
$1 = 0
```

This is the documented integer-divide-by-zero path: `iq_grow(q,
q->nalloc * 2)` is a no-op because `nlen == 0`, so `q->nalloc` stays `0`
and the modulo immediately faults. With the patch applied
(`iq_grow(q, capacity > 0 ? capacity : 1)`), `q->nalloc == 1` after
construction and the same driver runs to completion without crashing,
demonstrating that the public API now refuses to admit the zero-capacity
state.
