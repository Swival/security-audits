# 013 — `AsyncRequestWorkerFactor` out-of-range double-to-uint conversion

`set_worker_factor()` parses the directive as a `double` and assigns
`worker_factor = val * WORKER_FACTOR_SCALE` (16) into an `unsigned int`.
Configuring a value such that the scaled product exceeds `UINT_MAX`
(>= 268435456 on a 32-bit `unsigned int`) is undefined behavior.

## Setup

Build the event MPM with UBSan:

```bash
CFLAGS="-fsanitize=undefined -fno-sanitize-recover=all -g" \
    ./configure --with-mpm=event
make && make install
```

`httpd.conf`:

```apache
LoadModule mpm_event_module modules/mod_mpm_event.so

ServerLimit         1
StartServers        1
ThreadsPerChild     2
MaxRequestWorkers   2

AsyncRequestWorkerFactor 268435456
```

## Trigger

Just start httpd; the conversion runs during config parsing.

```bash
httpd -X
```

## Observed effect

UBSan diagnostic on stderr / `error_log`:

```text
server/mpm/event/event.c:4122:5: runtime error:
4.29497e+09 is outside the range of representable values of type 'unsigned int'
SUMMARY: UndefinedBehaviorSanitizer: undefined-behavior event.c:4122
```

On a non-sanitized build, the implementation-defined result of the
conversion makes `worker_factor` deterministically wrong. Print it from
gdb:

```text
(gdb) break event.c:4122
(gdb) run -X
(gdb) print val
$1 = 268435456
(gdb) next
(gdb) print worker_factor
$2 = 0          ; or another representation-bound clamp, depending on libc
```

A zero or otherwise truncated `worker_factor` then propagates into
`async_factor` and the pollset sizing in `setup_threads_runtime()` (see
finding 014), which is where the wrap becomes service-affecting.
