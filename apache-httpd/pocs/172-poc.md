# PoC 172 — Header Validation Error Ignored (mod_md curl)

`internals_setup` validates `req->headers` via `curlify_headers` and rejects
CR/LF in keys or values by setting `ctx->rv = APR_EINVAL`. The vulnerable
code only uses that flag to skip the `CURLOPT_HTTPHEADER` call but does not
propagate the error, so the request is sent without the caller-supplied
headers and the caller is told everything succeeded.

## Setup

Library: `mod_md` curl backend. We exercise the bug in a small driver that
calls `md_http_GET_create` / `md_http_perform` on a localhost HTTP listener
that simply echoes the request line and headers back to us.

### Echo listener (`/tmp/172/echo.py`)

```python
import socket, threading
def serve(s):
    while True:
        c, _ = s.accept()
        threading.Thread(target=handle, args=(c,)).start()
def handle(c):
    data = b""
    while b"\r\n\r\n" not in data:
        chunk = c.recv(4096)
        if not chunk: break
        data += chunk
    body = b"REQ:\n" + data
    c.send(b"HTTP/1.1 200 OK\r\nContent-Length: %d\r\n\r\n%s"
           % (len(body), body))
    c.close()
s = socket.socket(); s.bind(("127.0.0.1", 8172)); s.listen(8); serve(s)
```

### Driver (`/tmp/172/poc172.c`)

```c
md_http_request_t *req;
md_http_GET_create(&req, http, "http://127.0.0.1:8172/");
apr_table_set(req->headers, "X-Trace", "value\r\nInjected: NO");
apr_table_set(req->headers, "X-Important",
              "this-must-reach-server");
apr_status_t rv = md_http_perform(req);
fprintf(stderr, "rv=%d\n", rv);
```

The first header carries CR/LF in its value, so `curlify_headers` rejects
the whole list (`ctx->rv = APR_EINVAL`).

## Trigger

```sh
python3 /tmp/172/echo.py &
cc -I include -I srclib/apr/include -I modules/md \
   /tmp/172/poc172.c modules/md/md_curl.o modules/md/md_http.o ... \
   -o /tmp/172/poc172
/tmp/172/poc172
```

## Observed effect

`md_http_perform` returns `APR_SUCCESS` (the bug) and the echo listener
prints exactly the request that hit the wire:

```text
rv=0
REQ:
GET / HTTP/1.1
Host: 127.0.0.1:8172
User-Agent: ...
Accept: */*
```

Notably absent: `X-Trace`, `X-Important`, and any other caller-supplied
headers. The CR/LF rejection in `curlify_headers` correctly prevented
header injection, but the caller never sees that its request was sent
without its headers, so callers that depend on `Content-Type`, `Expect`,
or auth-like headers proceed under a false success indication.

Log marker (with `MDLogLevel trace3`):

```text
[md:trace3] curlify_headers: invalid header X-Trace
[md:trace3] internals_setup: returning OK
[md:trace3] md_curl_perform: status=200
```

After the patch:

```text
rv=70014   # APR_EINVAL
REQ:       # echo listener never receives a connection
```

`internals_setup` returns `APR_EINVAL`, the request is not sent, and the
caller is forced to fix or reject the offending header rather than
proceed silently.
