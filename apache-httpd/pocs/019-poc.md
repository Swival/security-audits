# 019 — SSL passphrase plaintext left in pool memory

`ssl_init_Module()` at `modules/ssl/ssl_engine_init.c:518` zeroes only the
pointer slots of the `pphrases` array, leaving the duplicated plaintext
passphrase strings intact in `ptemp` for the lifetime of the pool.

## Setup

Build httpd with `mod_ssl` against OpenSSL:

```bash
./configure --prefix=$PWD/inst --enable-ssl --with-ssl=/usr
make -j && make install
```

Generate an encrypted private key and matching cert:

```bash
openssl genrsa -aes256 -passout pass:hunter2supersecret -out key.pem 2048
openssl req -x509 -new -key key.pem -passin pass:hunter2supersecret \
    -subj '/CN=localhost' -days 30 -out cert.pem
```

`httpd.conf` snippet:

```apache
LoadModule ssl_module modules/mod_ssl.so
Listen 8443
SSLPassPhraseDialog builtin
<VirtualHost *:8443>
    ServerName localhost
    SSLEngine on
    SSLCertificateFile    /etc/httpd/cert.pem
    SSLCertificateKeyFile /etc/httpd/key.pem
</VirtualHost>
```

Run httpd in single-process mode so the parent pool persists for inspection:

```bash
./inst/bin/httpd -X -f /etc/httpd/conf/httpd.conf &
HTTPD_PID=$!
# enter passphrase: hunter2supersecret
```

## Trigger

After the startup banner reports `Init: Wiped out the queried pass phrases
from memory`, scan the live process heap for the plaintext:

```bash
gcore -o /tmp/httpd.core $HTTPD_PID
strings -n 12 /tmp/httpd.core.$HTTPD_PID | grep hunter2supersecret
```

Equivalent inspection with `gdb`:

```bash
gdb -p $HTTPD_PID -batch \
    -ex 'find /b 0x400000, +0x40000000, 0x68,0x75,0x6e,0x74,0x65,0x72,0x32'
```

## Observed effect

On the unpatched binary `strings` prints the passphrase verbatim, e.g.:

```text
hunter2supersecret
```

`gdb` `find` returns one or more hits inside the `ptemp` arena pages even
though the log claimed the passphrase was wiped. After applying the
`OPENSSL_cleanse()` patch, the same scans return no matches.

## Notes

The leak is bounded to the configuration pool lifetime, but the bytes can
also outlive the pool in unzeroed glibc free-list pages, so a process
core dump captured later still contains them on the unpatched build.
