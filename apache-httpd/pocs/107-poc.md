# PoC 107 — Partial Send Corrupts OCSP Request

`send_request()` advances the OCSP write pointer by the *requested* byte
count instead of the *actually sent* count, so any partial `apr_socket_send`
skips bytes and corrupts the outgoing OCSP HTTP request.

## Setup

The bug is in pure pointer arithmetic. The cleanest reproduction is a mock
OCSP responder whose accept socket forces small TCP receive windows so that
`apr_socket_send` returns a short write. We exercise the path through
`mod_ssl` OCSP stapling; the responder records what it actually receives.

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 4443
LoadModule ssl_module modules/mod_ssl.so
LoadModule socache_shmcb_module modules/mod_socache_shmcb.so

SSLEngine on
SSLCertificateFile  /tmp/srv.crt   # has Authority Information Access -> http://127.0.0.1:8888/
SSLCertificateKeyFile /tmp/srv.key
SSLUseStapling On
SSLStaplingCache shmcb:logs/stapling(65536)
SSLStaplingResponderTimeout 5
SSLStaplingForceURL http://127.0.0.1:8888/
```

`/tmp/srv.crt` is generated with:

```bash
openssl req -x509 -newkey rsa:2048 -nodes -days 30 \
  -keyout /tmp/srv.key -out /tmp/srv.crt \
  -subj '/CN=127.0.0.1' \
  -addext 'authorityInfoAccess=OCSP;URI:http://127.0.0.1:8888/'
```

Mock responder `/tmp/ocsp_stub.py` that forces partial reads on the server
side (which is how httpd observes a partial *send*: the kernel splits the
request into fragments because the responder reads slowly):

```python
import socket, threading, time, sys
srv = socket.socket()
srv.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
srv.bind(("127.0.0.1", 8888)); srv.listen(4)
print("listening")
while True:
    c, _ = srv.accept()
    c.setsockopt(socket.SOL_SOCKET, socket.SO_RCVBUF, 4096)
    buf = b""
    while True:
        chunk = c.recv(64)         # tiny reads -> client must do partial sends
        if not chunk: break
        buf += chunk
        time.sleep(0.05)           # back-pressure
    sys.stdout.write("RECV %d bytes: %r\n" % (len(buf), buf[:120]))
    sys.stdout.flush()
    c.sendall(b"HTTP/1.0 500 oops\r\n\r\n")
    c.close()
```

## Trigger

```bash
python3 /tmp/ocsp_stub.py &
httpd -X -f /tmp/httpd.conf &
sleep 1
# Ask for stapling
echo | openssl s_client -connect 127.0.0.1:4443 -status -servername 127.0.0.1 \
    -CAfile /tmp/srv.crt > /dev/null 2>&1
```

To make the partial send guaranteed-visible, optionally reduce the client
SO_SNDBUF inside `send_request` via `LD_PRELOAD`-overridden `apr_socket_send`,
or increase the OCSP request size by stapling a chain that produces a request
larger than the responder's receive window.

## Observed effect

The mock responder logs the request bytes httpd actually delivered. With the
buggy code, `RECV` shows a corrupted byte sequence: a contiguous middle slice
of the OCSP-over-HTTP request is missing and replaced by garbage from beyond
the original BIO chunk, e.g. `POST / HTTP/1.0\r\nHost: 127.0.0.1\r\n` followed
by random bytes where the OCSPRequest body should begin.

`logs/error_log` records OCSP failures along the lines of:

```text
[ssl:warn] AH01941: stapling_renew_response: responder error
```

With the patch (`wbuf += wlen;`), `RECV` shows a syntactically intact OCSP
request and `error_log` reports a normal stapling result.

## Marker

Mismatch between `Content-Length` advertised by httpd and the byte count the
responder receives, plus visible non-OCSP bytes inside the captured payload.
