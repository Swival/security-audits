# PoC 121 — Unsynchronized Thread Handle Close (WinNT MPM)

The Win32 `stderr_thread` HANDLE is closed by both
`service_stderr_thread()` and `mpm_nt_eventlog_stderr_flush()` without
synchronization, so the flush path can wait on or close a handle the
service path has already closed (or, worse, that has been recycled to an
unrelated kernel object).

## Setup

Windows-only. Build httpd 2.4.66 for Windows (`Makefile.win` /
`CMakeLists.txt`) and run it as a service via:

```bat
httpd -k install -n Apache24
sc start Apache24
```

Minimal `httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 8080
LoadModule mpm_winnt_module modules/mod_mpm_winnt.so
LoadModule unixd_module     modules/mod_unixd.so   # not on Win — omit
ErrorLog "logs/error.log"
```

Run httpd under Application Verifier with the *Handles* checker enabled to
make any double-close immediately fatal:

```bat
appverif -enable Handles -for httpd.exe
```

## Race driver

The race fires when `mpm_nt_eventlog_stderr_flush()` and
`service_stderr_thread()`'s exit cleanup overlap. We can force the overlap
by repeatedly stopping/starting the service while the eventlog thread is
mid-`ReadFile`, and by using a small driver that calls
`apr_terminate()` (which calls the flush) from another thread inside the
process.

A minimal in-process driver is the cleanest approach: load `httpd.exe`'s
mpm_winnt module under the debugger (`windbg`), break in
`mpm_nt_eventlog_stderr_flush`, and resume — meanwhile from a second
thread invoke `apr_pool_destroy()` of the parent pool to drive
`service_stderr_thread()` exit. A repro harness (compiled against the
mpm_winnt build) is:

```c
/* C:\src\race_driver.c */
#include <windows.h>
extern HANDLE stderr_thread;          /* mpm_winnt symbol */
extern void mpm_nt_eventlog_stderr_open(apr_pool_t *p);
extern void mpm_nt_eventlog_stderr_flush(void);

DWORD WINAPI flusher(LPVOID p) {
    Sleep(50);
    mpm_nt_eventlog_stderr_flush();   /* race A */
    return 0;
}

int main(void) {
    apr_pool_t *p;
    apr_initialize();
    apr_pool_create(&p, NULL);
    mpm_nt_eventlog_stderr_open(p);
    HANDLE t = CreateThread(NULL, 0, flusher, NULL, 0, NULL);
    /* Slam the inherited stderr to make service_stderr_thread leave its
       ReadFile loop and run its own CloseHandle(stderr_thread). */
    CloseHandle(GetStdHandle(STD_ERROR_HANDLE));   /* race B */
    WaitForSingleObject(t, INFINITE);
    return 0;
}
```

Build and run repeatedly:

```bat
cl race_driver.c /link httpd.lib libhttpd.lib
for /L %i in (1,1,200) do race_driver.exe
```

## Trigger

```bat
race_driver.exe
```

(repeat under Application Verifier).

## Observed effect

On a vulnerable build, one of the iterations produces:

```text
VERIFIER STOP 0000000000000300: pid 0x12AC: Invalid handle exception for current stack trace.
    HANDLE value = 0x000000000000019C
    ...
    nt!CloseHandle
    server!mpm_nt_eventlog_stderr_flush+0x52
```

or, without the verifier:

```text
[mpm_winnt:crit] (Invalid handle) WaitForSingleObject failed
```

and intermittently a SIGABRT-like fault when the recycled handle happens
to be a kernel object owned by another component.

After applying the patch (`InterlockedExchangePointer` claim of
`stderr_thread` and removal of the close inside `service_stderr_thread`),
running the same harness 200 times produces zero handle violations.

## Marker

`VERIFIER STOP ... Invalid handle ... CloseHandle` with the top frame
inside `mpm_nt_eventlog_stderr_flush` (`server/mpm/winnt/nt_eventlog.c:109`).

## Platform note

Windows-specific. There is no equivalent path on POSIX MPMs: the race only
exists for the Win32 `HANDLE`-based shutdown sequence.
