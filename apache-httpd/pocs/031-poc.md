# 031 — mod_include leaks subrequest pools per SSI include

`handle_include()` at `modules/filters/mod_include.c:1455` (and 1880,
1899) deliberately skips `ap_destroy_sub_req()` for every SSI include.
Each include retains its own subrequest pool until the parent request
completes, so a single SSI document with many includes accumulates
proportional memory.

## Setup

Build httpd with mod_include:

```bash
./configure --prefix=$PWD/inst --enable-include
make -j && make install
```

`httpd.conf` snippet:

```apache
LoadModule include_module modules/mod_include.so

DocumentRoot /var/www
<Directory "/var/www">
    Options +Includes
    AddType text/html .shtml
    AddOutputFilter INCLUDES .shtml
    Require all granted
</Directory>
```

A small target document for the includes to reference:

```bash
mkdir -p /var/www
printf 'x' > /var/www/leaf.html
```

A driver that emits an SSI document with many `<!--#include virtual=-->`
tags:

```bash
python3 - >/var/www/big.shtml <<'PY'
import sys
N = 20000
sys.stdout = open('/var/www/big.shtml','w')
print('<html><body>')
for i in range(N):
    print(f'<!--#include virtual="/leaf.html?{i}" -->')
print('</body></html>')
PY
```

Run httpd in single-process mode so memory growth is observable on one PID:

```bash
./inst/bin/httpd -X -f /etc/httpd/conf/httpd.conf &
HTTPD_PID=$!
```

## Trigger

```bash
RSS_BEFORE=$(awk '/VmRSS/{print $2}' /proc/$HTTPD_PID/status)
echo "RSS before: $RSS_BEFORE kB"

curl -s http://127.0.0.1:8080/big.shtml > /dev/null

RSS_AFTER=$(awk '/VmRSS/{print $2}' /proc/$HTTPD_PID/status)
echo "RSS after:  $RSS_AFTER kB"
echo "Delta:      $((RSS_AFTER - RSS_BEFORE)) kB"
```

For a finer-grained view, run httpd under `valgrind --tool=massif` and
snapshot during request processing:

```bash
valgrind --tool=massif --pages-as-heap=yes \
    ./inst/bin/httpd -X -f conf/httpd.conf &
curl -s http://127.0.0.1:8080/big.shtml > /dev/null
ms_print massif.out.* | less
```

## Observed effect

Unpatched: RSS grows by tens of MB after a single request, scaling roughly
linearly with the number of include tags. Massif attributes the growth to
`make_sub_request` -> `apr_pool_create` called from `handle_include`. The
allocation is not freed until the main request pool is destroyed at the
end of the request — but it stays alive for the entire response, so a
streaming client that reads slowly extends the leak indefinitely.

Patched: RSS returns to its pre-request baseline within the response, and
Massif shows `ap_destroy_sub_req` reclaiming each subrequest pool.

## Notes

The leak amplifies with attacker-controlled SSI documents — for example,
user-uploaded `.shtml` on shared hosting, or an attacker-controlled origin
behind a reverse proxy that buffers SSI. A slow reader plus 50k includes
pins ~hundreds of MB per worker until the request finishes.
