# PoC 051 — racy `requests_this_child` counter on Windows MPM

`worker_main()` increments a static `requests_this_child` non-atomically
across threads, so concurrent connections can lose updates and fail to
trigger `MaxConnectionsPerChild` recycle.

## Setup

Platform: Windows (the bug is in `server/mpm/winnt/child.c`). Build httpd
with the WinNT MPM (Visual Studio or MinGW), no patch applied.

Minimum `httpd.conf`:

```
Listen 127.0.0.1:8080
ServerName localhost
ThreadsPerChild 64
MaxConnectionsPerChild 1
LoadModule mpm_winnt_module modules/mod_mpm_winnt.so
DocumentRoot "C:/Apache24/htdocs"
```

`MaxConnectionsPerChild 1` means the very first served request should make
the child signal the parent for restart. With 64 worker threads the lost
update is easy to observe.

## Trigger

Drive 64 concurrent connections from a second host or another process so
all worker threads run `requests_this_child++` simultaneously:

```
ab -n 256 -c 64 http://127.0.0.1:8080/
```

Race driver (PowerShell) that fires bursts repeatedly:

```
1..50 | ForEach-Object -Parallel {
    & ab.exe -n 64 -c 64 http://127.0.0.1:8080/ | Out-Null
} -ThrottleLimit 50
```

Observe the child PID across bursts:

```
Get-CimInstance Win32_Process -Filter "Name='httpd.exe'" |
    Select-Object ProcessId, CreationDate
```

## Observed effect

With `MaxConnectionsPerChild 1`, expectation: child PID changes after every
served request. Pre-patch, the child PID survives across many handled
connections because `requests_this_child > ap_max_requests_per_child`
evaluates false when two threads both store the value `1`.

`error.log` should contain one `AH00428 Child: Maximum number of allowed
connections reached. Sending parent the shutdown signal.` per recycle.
Counting that line versus the actual number of served requests shows
recycle messages missed:

```
findstr /c:"AH00428" logs\error.log | find /c /v ""
```

is strictly less than the `ab` request count when the race fires. With the
atomic patch applied, the two counts converge.

## Note

A pure user-space repro on Linux is possible by lifting `worker_main()`'s
counter logic into a tiny harness compiled with `-fsanitize=thread`. TSan
flags the race directly on the `requests_this_child++` site.
