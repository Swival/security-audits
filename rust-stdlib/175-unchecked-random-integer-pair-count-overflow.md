# Unchecked Random Integer Pair Count Overflow

## Classification

Data integrity bug, medium severity.

Confidence: certain.

## Affected Locations

- `library/compiler-builtins/libm-test/src/generate/random.rs:150`
- `library/compiler-builtins/libm-test/src/generate/random.rs:166`

## Summary

Integer-pair random test case generators computed their expected case count with unchecked `u64` multiplication. For large `iteration_count` values, `count0 * count1` overflows in optimized/release builds with overflow checks disabled, corrupting the count passed to `KnownSize`.

The actual iterator still represents the full Cartesian product, but the reported expected length can wrap to an unrelated value, including `0`.

## Provenance

Reported by Swival Security Scanner: https://swival.dev

The finding was reproduced and confirmed against `library/compiler-builtins/libm-test/src/generate/random.rs`.

## Preconditions

- `iteration_count(ctx, 0)` and `iteration_count(ctx, 1)` return sufficiently large `u64` values.
- A two-integer `RandomInput` implementation generated by `impl_random_input_int` is used.
- The build permits arithmetic overflow wrapping, such as optimized/release builds with overflow checks disabled.

## Proof

The integer pair implementations generated by `impl_random_input_int` read both counts from `iteration_count`:

- `count0 = iteration_count(ctx, 0)`
- `count1 = iteration_count(ctx, 1)`

They then build a Cartesian product iterator with `product2(iter0, iter1)`, but previously returned the expected count as:

```rust
(iter, count0 * count1)
```

For values such as:

```rust
count0 = 4_294_967_296
count1 = 4_294_967_296
```

the mathematical product is `2^64`, which does not fit in `u64`. In wrapping arithmetic, this becomes `0`.

That wrapped count is propagated into `KnownSize::new` from `get_test_cases`, causing `KnownSize::size_hint` to report an incorrect remaining length. The end assertion cannot detect the mismatch until the enormous real product iterator is exhausted.

Float-pair generators already used `strict_mul`, so the defect was localized to integer-pair random generators.

## Why This Is A Real Bug

The returned count is metadata consumers rely on for exact iterator sizing, progress tracking, accounting, and validation. When the count wraps, the iterator and its reported size diverge.

This is not merely a theoretical arithmetic issue: the affected path is reachable through integer pair cases generated by `impl_random_input_int`, and the overflowed value is directly used by `KnownSize`.

## Fix Requirement

Replace unchecked multiplication with checked or strict arithmetic so overflow is detected immediately instead of silently corrupting the expected count.

Acceptable fixes include:

```rust
count0.strict_mul(count1)
```

or an equivalent `checked_mul(...).expect(...)`.

## Patch Rationale

The patch changes both unchecked integer-pair multiplications to `strict_mul`:

```diff
-                (iter, count0 * count1)
+                (iter, count0.strict_mul(count1))
```

This matches the existing behavior of float-pair and mixed integer/float random generators in the same file, which already use `strict_mul` for expected count computation.

The change preserves behavior for valid products and converts overflow from silent metadata corruption into an immediate panic.

None
