# order-two torsion passes subgroup check

## Classification

Security control failure. Severity: critical. Confidence: certain.

## Affected Locations

- `lib/std/crypto/25519/edwards25519.zig:112`
- Function: `Edwards25519.rejectIdentity`
- Caller affected: `Edwards25519.rejectUnexpectedSubgroup`

## Summary

`rejectUnexpectedSubgroup()` is intended to reject Edwards25519 points outside the prime-order subgroup. It fails open for points with an order-two torsion component because `rejectIdentity()` identifies the neutral element using only `x == 0`.

On Edwards25519, the neutral element in extended coordinates is specifically `x = 0` and `y = z`. The order-two torsion point also has `x = 0`, but is not the neutral element. As a result, subgroup-order multiplication of a malicious point `P + T` can produce the order-two torsion component `T`, which is incorrectly treated as identity, causing the subgroup check to return success.

## Provenance

Verified by Swival.dev Security Scanner: https://swival.dev

## Preconditions

- Caller invokes `rejectUnexpectedSubgroup()` on an attacker-supplied Edwards25519 point.

## Proof

A malicious point `P + T`, where `P` is in the prime-order subgroup and `T` is an order-two torsion point, encodes as:

```text
9599999999999999999999999999999999999999999999999999999999999999
```

Observed propagation:

1. Attacker supplies encoded point `9599...9999`.
2. `Edwards25519.fromBytes()` accepts it as a valid curve point.
3. `rejectUnexpectedSubgroup()` calls `rejectLowOrder()`.
4. `rejectLowOrder()` does not reject because the point is not purely low-order.
5. `rejectUnexpectedSubgroup()` multiplies the point by the prime subgroup order.
6. Since the subgroup order is odd, `[subgroup_order](P + T) = T`.
7. `T` is the order-two torsion point with `x = 0`.
8. `rejectIdentity()` checks only `p.x.isZero()`, incorrectly classifies `T` as the neutral element, and returns `error.IdentityElement`.
9. `rejectUnexpectedSubgroup()` catches that error and returns success.

The reproduced Zig test confirmed that `rejectUnexpectedSubgroup()` returns success instead of `error.UnexpectedSubgroup` for the encoded malicious point.

## Why This Is A Real Bug

`rejectUnexpectedSubgroup()` is a prime-subgroup membership security control. It should accept only points in the subgroup generated by the Edwards25519 base point.

The vulnerable implementation accepts a point outside that subgroup. The accepted point has a non-zero order-two torsion component, and subtracting the base point yields the low-order order-two point. This is a deterministic fail-open in subgroup validation, not a cosmetic or unreachable issue.

## Fix Requirement

Reject the neutral element using the full extended-coordinate neutral-element condition, not the X coordinate alone.

For Edwards25519 extended coordinates, the neutral element is represented by:

```text
x = 0
y = z
```

Therefore `rejectIdentity()` must require both `x == 0` and `y == z`.

## Patch Rationale

The patch changes `rejectIdentity()` from:

```zig
if (p.x.isZero()) {
```

to:

```zig
if (p.x.isZero() and p.y.sub(p.z).isZero()) {
```

This preserves rejection of the actual neutral element while no longer misclassifying the order-two torsion point as identity. Consequently, when `rejectUnexpectedSubgroup()` computes `[subgroup_order]p = T`, `rejectIdentity()` no longer errors, and `rejectUnexpectedSubgroup()` correctly returns `error.UnexpectedSubgroup`.

None
